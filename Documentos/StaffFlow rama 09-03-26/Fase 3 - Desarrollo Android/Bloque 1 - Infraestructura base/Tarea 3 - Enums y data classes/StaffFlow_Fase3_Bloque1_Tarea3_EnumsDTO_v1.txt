Este fichero documenta la creacion de los enums del dominio y las
data classes Kotlin (requests y responses) que representan el contrato
entre la app Android y la API REST del backend de StaffFlow.

====================================================
1. CONTEXTO
====================================================

Con la estructura de paquetes ya definida (Tarea 2), el siguiente
paso es crear los tipos de datos que toda la capa de red va a
necesitar. Estos tipos son la traduccion directa al mundo Kotlin de:

  - Los enums Java del backend (domain/enums/)
  - Los DTOs Java del backend (dto/request/ y dto/response/)

Gson los usa para deserializar automaticamente las respuestas JSON
de la API y para serializar los cuerpos de las peticiones. No requieren
ninguna anotacion especial porque los nombres de campo coinciden
exactamente con los del backend (convencion acordada en el diseno).

====================================================
2. ENUMS CREADOS (domain/model/)

Se han creado 6 ficheros, uno por enum, bajo el paquete

           PERMISO_RETRIBUIDO, DIA_LIBRE_COMPENSATORIO, DIA_LIBRE
  Uso: AusenciaResponse, AusenciaRequest (E30, E33, E34).

TipoPausa.kt
  Valores: COMIDA, DESCANSO, AUSENCIA_RETRIBUIDA, OTROS

  Valores: OPERARIO, ADMINISTRATIVO, TECNICO, ENCARGADO, OTRO

       en P17 (ParteDiarioFragment) y P12 (MiHoyFragment).

3. DATA CLASSES CREADAS (data/remote/dto/)
====================================================

Se han creado 2 ficheros bajo com.staffflow.android.data.remote.dto:

----------------------------------------------------
3.1 Responses.kt -- 16 data classes
----------------------------------------------------

LoginResponse        <- E01. Contiene token, rol, username, empleadoId
                        (null si ADMIN).
UsuarioResponse      <- E02, E09, E10. Sin datos sensibles.
EmpleadoResponse     <- E14, E15, E21. Sin pinTerminal por seguridad.
                        numeroEmpleado formato EMP-001 (D-030).
EmpresaResponse      <- E06. Singleton id=1.
FichajeResponse      <- E22-E26. horaEntrada/horaSalida nullable.
PausaResponse        <- E27-E29. horaFin/duracionMinutos nullable
                        mientras la pausa esta activa.
AusenciaResponse     <- E30-E34. empleadoId null = festivo global.
                        procesado=true bloquea modificacion y borrado.
SaldoResponse        <- E38, E39, E41. Con 3 nested data classes:
                        VacacionesDesglose, AsuntosPropiosDesglose,
                        HorasDesglose. saldoHoras positivo/negativo
                        se muestra en verde/rojo (P09, P26).
ParteDiarioResponse  <- E35. Contiene List<DetallePresenciaResponse>
                        y contadores de resumen para los chips de P17.
DetallePresenciaResponse <- E35, E37. Estado con color en P17 y P12.
SinJustificarResponse    <- E36. Listado en P18.
MensajeResponse      <- Respuesta generica de confirmacion.
ErrorResponse        <- Estructura estandar de error del backend.
                        Todos los 4xx/5xx usan este formato.
TerminalEntradaResponse  <- E48. Mostrada en P06 (3 segundos).
TerminalSalidaResponse   <- E49. Mostrada en P06 (3 segundos).
TerminalPausaResponse    <- E50, E51. horaInicioPausa null en E51.
                            duracionPausaMinutos null en E50.

----------------------------------------------------
3.2 Requests.kt -- 16 data classes
----------------------------------------------------

LoginRequest             <- E01.
PasswordChangeRequest    <- E03.
PasswordRecoveryRequest  <- E04. Backend siempre 200 (anti-enumeracion).
PasswordResetRequest     <- E05. Error 400 si token expirado.
UsuarioRequest           <- E08. Password en claro; BCrypt en backend.
UsuarioPatchRequest      <- E11. Todos los campos nullable.
EmpleadoRequest          <- E13. pinTerminal exactamente 4 digitos.
EmpleadoPatchRequest     <- E16. Todos los campos nullable.
EmpresaRequest           <- E07. PUT completo al singleton.
FichajeRequest           <- E22. observaciones obligatoria (RNF-L02).
FichajePatchRequest      <- E23. observaciones OBLIGATORIA aunque

PausaRequest             <- E27. horaFin null = pausa activa.
PausaPatchRequest        <- E28. Todos los campos nullable.
AusenciaRequest          <- E30. empleadoId null = festivo global.
AusenciaPatchRequest     <- E31. Todos los campos nullable.
TerminalPinRequest       <- E48, E49, E51. pin 4 digitos +
                            dispositivoId para mecanismo de bloqueo
                            por intentos fallidos (HTTP 423).
TerminalPausaRequest     <- E50. Extiende pin con tipoPausa.

====================================================
4. DECISIONES DE IMPLEMENTACION
====================================================

Decision 1 -- Un enum por fichero:
  Cada enum en su propio fichero .kt bajo domain/model. Facilita
  la navegacion en el IDE y es consistente con el estilo del proyecto.

Decision 2 -- Dos ficheros para DTOs (Responses.kt y Requests.kt):
  Se agrupan en dos ficheros en lugar de uno por clase. Los DTOs
  son tipos de datos simples sin logica; separarlos en dos ficheros
  (responses/requests) es suficiente para la navegacion sin crear
  proliferacion de ficheros de una sola clase.

Decision 3 -- FichajePatchRequest.observaciones no es nullable:
  Segun RNF-L02, las observaciones son obligatorias en crear Y
  modificar fichajes. Se refleja en el tipo: String, no String?.
  El formulario P20 debe validar este campo antes de enviar.

Decision 4 -- SaldoResponse con nested data classes:
  Se mantiene la misma estructura anidada que el backend Java para
  que Gson deserialice correctamente el JSON sin configuracion extra.
