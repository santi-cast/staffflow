Este fichero documenta la estructura de paquetes definida para
la aplicacion Android de StaffFlow y la razon de cada decision
de organizacion.

====================================================
1. CONTEXTO
====================================================

El proyecto tenia una estructura parcial creada en la Fase 0
del proyecto (anterior a esta sesion):
  data/remote/api      (tenia NetworkModule.kt)
  data/remote/dto      (vacio)
  data/remote/repository (vacio)
  domain/model         (vacio)
  ui/fichaje           (vacio)
  ui/login             (vacio)
  util                 (vacio)

La Tarea 2 completa esta estructura anadiendo los paquetes que
faltaban y tomando dos decisiones de nomenclatura respecto a la
propuesta inicial.

====================================================
2. DECISIONES DE NOMENCLATURA
====================================================

Decision 1 -- domain/model en lugar de model a secas:
  El backend de StaffFlow organiza sus enums bajo el paquete
  domain. Se mantiene la misma convencion en Android para
  consistencia entre capas del proyecto.

Decision 2 -- ui/fichaje y ui/login en lugar de ui/terminal y ui/auth:
  Los nombres fichaje y login son mas descriptivos del contenido
  real de cada paquete en el contexto de StaffFlow y se
  prefieren sobre los nombres tecnicos terminal y auth.

====================================================
3. ESTRUCTURA FINAL DE PAQUETES
====================================================

com.staffflow.android
  data/
    local/             <- DataStore: sesion persistente (token, rol,
                          username, empleadoId)
    remote/
      api/             <- NetworkModule + interfaces Retrofit por grupo
                          de endpoints
      dto/             <- data classes Kotlin: responses y request bodies
                          que Gson deserializa de la API
      repository/      <- repositorios que encapsulan las llamadas
                          Retrofit y exponen metodos suspendibles
                          a los ViewModels
  domain/
    model/             <- enums del dominio: Rol, TipoFichaje,
                          TipoAusencia, TipoPausa, CategoriaEmpleado,
                          EstadoPresencia
  ui/
    admin/             <- fragments exclusivos ADMIN: P31 Usuarios,
                          P32 FormUsuario, P34 Empresa
    empleado/          <- fragments exclusivos EMPLEADO: P08 MiPerfil,
                          P09 MiSaldo, P10 MisFichajes,
                          P11 MisAusencias, P12 MiHoy
    encargado/         <- fragments ENCARGADO y ADMIN: P17 ParteDiario,
                          P18 SinJustificar, P19 Fichajes,
                          P20 FormFichaje, P23 Ausencias,
                          P24 FormAusencia, P26 SaldoIndividual,
                          P27 SaldosGlobales, P28 Informes
    fichaje/           <- flujo terminal sin JWT: P01 Terminal,
                          P06 Confirmacion, P07 TipoPausa
    login/             <- flujo autenticacion: P02 Login, P03 Recovery,
                          P04 CambiarPassword, P05 ResetPassword
    shared/            <- fragments compartidos ADMIN y ENCARGADO:
                          P13 Empleados, P14 DetalleEmpleado,
                          P15 FormEmpleado
  util/                <- constantes, extensiones Kotlin, helpers
                          de formato de fecha y otros auxiliares
