1. CONTEXTO
====================================================

La Tarea 3 del Bloque 6 implementa el Grupo 9 de la API REST:
consulta y recalculo de saldos anuales de vacaciones, asuntos propios
y horas. Cubre los requisitos RF-35, RF-36, RF-37 y RF-53.

Antes de escribir codigo se revisaron los contratos de E38-E41 en
StaffFlow_Endpoints_v3.docx y se leyeron las entidades SaldoAnual y
Empleado, y los repositorios SaldoAnualRepository y EmpleadoRepository
para conocer los nombres exactos de campos y metodos existentes.

El grupo tiene una particularidad importante: la tabla saldos_anuales
no se rellena por peticion del usuario sino por el proceso nocturno
@Scheduled (ProcesoDiario, Tarea 4). En el momento de implementar
esta tarea la tabla puede estar vacia. Por eso E40 (recalcular)
implementa el patron findOrCreate: si no existe el registro de saldo
para el año solicitado, lo crea con los valores iniciales del contrato
del empleado antes de recalcular. Esto permite usar E40 para generar
el primer registro de un empleado sin depender del proceso nocturno.

Decision de diseno previa al escribir codigo:

  Calculo de horas en memoria sin query adicional a FichajeRepository
  (para E38, E39 y E41):
    esperadas  = diasTrabajados * jornadaDiariaMinutos / 60.0
    trabajadas = esperadas + saldoHoras

  La alternativa era sumar jornadaEfectivaMinutos de todos los fichajes
  del año desde FichajeRepository, lo que implicaba una query adicional
  por empleado. Se descarto porque saldoHoras ya acumula la diferencia
  real fichada por el proceso nocturno de forma exacta, haciendo
  innecesaria la query adicional.

  La formula funciona porque saldoHoras = trabajadas - esperadas, por
  lo que despejar trabajadas = esperadas + saldoHoras es exacto.
  Ejemplo: 50 dias trabajados, jornada 480 min/dia, saldoHoras = +2h:
    esperadas  = 50 * 480 / 60 = 400h
    trabajadas = 400 + 2       = 402h

  E40 (recalcular) si usa FichajeRepository porque recalcula desde
  cero iterando todos los fichajes del año, reiniciando los contadores
  antes de sumar para garantizar la idempotencia.

E38 -- GET /api/v1/saldos/{empleadoId}
RF-35 (saldo anual individual)
----------------------------------------------------

Roles: ADMIN, ENCARGADO
Autenticacion: JWT requerido

Path params:
  {empleadoId} -- ID del empleado

Query params:
  ?anio=number   opcional -- por defecto ano actual (Year.now())

Respuesta 200:
  {
    "empleadoId": n,
    "anio": 2026,
    "vacaciones": {
      "derechoAnio": 22,
      "pendientesAnterior": 0,
      "consumidos": 0,
      "disponibles": 22
    },
    "asuntosPropios": {
      "derechoAnio": 3,
      "pendientesAnterior": 0,
      "consumidos": 0,
      "disponibles": 3
    },
    "horas": {
      "esperadas": 400.00,
      "trabajadas": 402.00,
      "saldoHoras": 2.00
    },
    "calculadoHastaFecha": "2026-03-13"
  }

Logica del service:
  1. findById(empleadoId) --> 404 si el empleado no existe.
  2. findByEmpleadoIdAndAnio(empleadoId, anio) --> 404 si no hay
     registro de saldo para ese ano.
  3. toSaldoResponse(saldo, empleado) --> calcula horas en memoria
     y construye el DTO.

Codigos HTTP:
  200 OK   saldo anual completo
  403      rol insuficiente
  404      empleado no encontrado o sin saldo para ese ano

E39 -- GET /api/v1/saldos
RF-36 (saldo anual global)
----------------------------------------------------

Roles: ADMIN, ENCARGADO
Autenticacion: JWT requerido

Query params:
  ?anio=number   opcional -- por defecto ano actual

Respuesta 200:
  [ { saldo completo por empleado }, ... ]
  Lista vacia si no hay saldos registrados para ese ano.

Logica del service:
  1. findByAnio(anio) --> lista de todos los saldos de ese ano.
     No filtra por activo: un empleado inactivo puede tener saldo
     historico valido consultable.
  2. Por cada saldo: toSaldoResponse(s, s.getEmpleado()).
     Nota: con @ManyToOne fetch LAZY Hibernate hara una query
     adicional por empleado al acceder a s.getEmpleado() (N+1).
     Para el volumen de una PYME (max ~50 empleados) es aceptable.
     Si se detecta problema de rendimiento la solucion es anadir
     @EntityGraph o JOIN FETCH en findByAnio.
  3. Devuelve List<SaldoResponse>.

Codigos HTTP:
  200 OK   lista de saldos (puede ser vacia)
  403      rol insuficiente

E40 -- POST /api/v1/saldos/{empleadoId}/recalcular
RF-37 (recalcular saldo)
----------------------------------------------------

Roles: solo ADMIN
Autenticacion: JWT requerido

Path params:
  {empleadoId} -- ID del empleado

Query params:
  ?anio=number   opcional -- por defecto ano actual

Respuesta 200:
  { saldo recalculado completo -- mismo formato que E38 }

Logica del service:
  1. findById(empleadoId) --> 404 si el empleado no existe.
  2. findOrCreate: findByEmpleadoIdAndAnio() -- si no existe el
     registro lo crea con crearSaldoInicial() con los valores
     del contrato del empleado (diasVacacionesAnuales,
     diasAsuntosPropiosAnuales, todos los contadores a 0).
  3. Obtiene todos los fichajes del ano con
     findByEmpleadoIdAndFechaBetween(id, 01/01/anio, 31/12/anio).
  4. Reinicia contadores a 0 (garantia de idempotencia: siempre
     parte de cero para que ejecutar N veces de el mismo resultado).
  5. Itera los fichajes acumulando segun TipoFichaje:
       NORMAL / FESTIVO_NACIONAL / FESTIVO_LOCAL /
       DIA_LIBRE_COMPENSATORIO --> diasTrabajados++ y acumula
         diferencia de jornada en saldoHoras:
         diferencia = jornadaEfectivaMinutos - jornadaDiariaMinutos
         (positivo = extra, negativo = deficit)
       VACACIONES --> diasVacacionesConsumidos++
       ASUNTO_PROPIO --> diasAsuntosPropiosConsumidos++
       PERMISO_RETRIBUIDO --> diasPermisoRetribuido++
       BAJA_MEDICA --> diasBajaMedica++
       AUSENCIA_INJUSTIFICADA --> diasAusenciaInjustificada++
  6. Recalcula disponibles:
       diasVacacionesDisponibles = derechoAnio + pendientesAnterior
                                   - consumidos
       diasAsuntosPropiosDisponibles = idem para asuntos propios
  7. Actualiza calculadoHastaFecha = LocalDate.now().
  8. Guarda y devuelve toSaldoResponse(saldo, empleado).

Por que POST y no PATCH:
  La convencion PUT/PATCH del proyecto reserva PATCH para cambios
  de estado o campos parciales. E40 no modifica campos parciales:
  recalcula el registro completo desde cero. POST es el verbo
  correcto para una operacion que ejecuta un proceso completo
  (convencion cerrada en Fase 1, Endpoints_v3.docx).

Codigos HTTP:
  200 OK   saldo recalculado completo
  403      rol insuficiente (solo ADMIN)
  404      empleado no encontrado

E41 -- GET /api/v1/saldos/me
RF-53 (saldo propio del empleado autenticado)
----------------------------------------------------

Roles: solo EMPLEADO
Autenticacion: JWT requerido

Query params:
  ?anio=number   opcional -- por defecto ano actual

Respuesta 200:
  { mismo desglose completo que E38 }

Logica del service:
  1. authentication.getName() --> username --> findByUsuarioUsername()
     --> empleado. Si ADMIN llama a este endpoint Spring devuelve
     403 antes de llegar aqui por @PreAuthorize("hasRole('EMPLEADO')").
  2. findByEmpleadoIdAndAnio(empleado.getId(), anio) --> 404 si no
     hay saldo registrado para ese ano. Puede ocurrir si el proceso
     nocturno aun no ha creado el registro del ano en curso.
  3. toSaldoResponse(saldo, empleado) --> mismo calculo que E38.

Por que no se usa authentication.getDetails() para obtener empleadoId:
  getDetails() devuelve WebAuthenticationDetails (IP y sessionId),
  no los claims del JWT. Los claims del token no estan disponibles
  directamente en el objeto Authentication de Spring Security sin
  implementacion adicional. La solucion correcta es el patron
  authentication.getName() --> username --> findByUsuarioUsername(),
  que ya usan E34 (AusenciaService) y E37 (PresenciaService) con
  el mismo proposito. Ningun controller accede directo a un repository
  (regla de oro del proyecto).

Convencion /me primero en el controller:
  E41 (/me) se declara ANTES que E38 (/{empleadoId}) en el controller.
  Si se declarara despues, Spring intentaria resolver la cadena "me"
  como un Long al hacer el match del path /{empleadoId}, lanzando
  un error de conversion de tipos en runtime. Mismo patron aplicado
  en EmpleadoController (E21), FichajeController (E26),
  AusenciaController (E34) y PresenciaController (E37).

Codigos HTTP:
  200 OK   saldo anual del empleado autenticado
  403      acceso denegado (rol incorrecto)
  404      sin saldo registrado para ese ano

4. REPOSITORY -- METODOS CUSTOM ANADIDOS
====================================================

SaldoAnualRepository -- 1 metodo nuevo:

  findByAnio(Integer anio)
    Sin @Query explicita. Spring Data JPA genera la implementacion
    SQL automaticamente a partir del nombre del metodo.
    Devuelve todos los registros de saldos_anuales donde anio = :anio.
    No filtra por activo porque un empleado inactivo puede tener
    saldo historico valido. Usado exclusivamente por E39 listarTodos().

    Por que no se usa @Query explicita:
      El metodo accede a un unico campo sin condiciones adicionales.
      Spring Data infiere correctamente SELECT s FROM SaldoAnual s
      WHERE s.anio = :anio. Solo se usa @Query cuando Spring Data
      no puede inferir la query (condiciones OR, JOIN FETCH, etc.),
      como ocurre en buscarPorTexto (EmpleadoRepository) o en los
      metodos de FichajeRepository y PausaRepository.

SaldoAnualRepository -- metodos preexistentes usados:

  findByEmpleadoIdAndAnio(Long empleadoId, Integer anio)
    Preexistente del esqueleto del Bloque 1.
    Usado por E38, E41 (consulta) y E40 (findOrCreate).
    Devuelve Optional: vacio = 404 en E38/E41, o trigger de
    crearSaldoInicial() en E40.

  findByEmpleadoId(Long empleadoId)
    Preexistente del esqueleto del Bloque 1.
    No se usa en esta tarea. Reservado para CierreAnual (decision n12).

5. DTO -- ESTRUCTURA DE SALDORESPONSE
====================================================

SaldoResponse usa clases estaticas internas en lugar de ficheros
separados porque los tres desgloses solo tienen sentido dentro
del contexto de SaldoResponse y no se reutilizan en ningun otro DTO.

Alternativa descartada: crear VacacionesDesglose.java,
AsuntosPropiosDesglose.java y HorasDesglose.java como ficheros
independientes en dto.response. Aniadira 3 ficheros al proyecto
sin ningun beneficio real, ya que ninguno de esos DTOs se usa
fuera de SaldoResponse.

Estructura de la clase:

  SaldoResponse (@Data @AllArgsConstructor)
    Long empleadoId
    Integer anio
    VacacionesDesglose vacaciones
    AsuntosPropiosDesglose asuntosPropios
    HorasDesglose horas
    LocalDate calculadoHastaFecha

  VacacionesDesglose (@Data @AllArgsConstructor -- static)
    Integer derechoAnio          -- Empleado.diasVacacionesAnuales
    Integer pendientesAnterior   -- arrastrados del ano anterior
    Integer consumidos           -- consumidos este ano
    Integer disponibles          -- derechoAnio + pendientes - consumidos

  AsuntosPropiosDesglose (@Data @AllArgsConstructor -- static)
    Integer derechoAnio          -- Empleado.diasAsuntosPropiosAnuales
    Integer pendientesAnterior   -- arrastrados del ano anterior
    Integer consumidos           -- consumidos este ano
    Integer disponibles          -- derechoAnio + pendientes - consumidos

  HorasDesglose (@Data @AllArgsConstructor -- static)
    BigDecimal esperadas         -- diasTrabajados * jornadaDiariaMin / 60
    BigDecimal trabajadas        -- esperadas + saldoHoras
    BigDecimal saldoHoras        -- persistido en SaldoAnual

Por que @Data + @AllArgsConstructor sin @Builder:
  Patron del proyecto establecido en Bloque 2 y mantenido en todos
  los DTOs response: LoginResponse, MensajeResponse, EmpleadoResponse,
  etc. Todos usan new XxxResponse(...) con el constructor de todos
  los argumentos. @Builder se descarto porque anade verbosidad sin
  beneficio real para objetos que siempre se construyen completos.

Por que BigDecimal para las horas:
  Decision n22 del proyecto: BigDecimal para cualquier campo con
  precision decimal en calculos de jornada. Un empleado con jornada
  de 7.5h diarias y saldo de +0.25h daria un resultado incorrecto
  con double por la imprecision del punto flotante binario.
  BigDecimal garantiza la precision exacta del calculo.

====================================================
6. DECISIONES TECNICAS DE ESTA TAREA
====================================================

Decision T3-A -- Calculo de horas en memoria sin query adicional
  Para E38, E39 y E41 el bloque horas del DTO se calcula en memoria:
    esperadas  = diasTrabajados * jornadaDiariaMinutos / 60.0
    trabajadas = esperadas + saldoHoras
  Alternativa descartada: sumar jornadaEfectivaMinutos de todos los
  fichajes del ano con una query adicional a FichajeRepository.
  La alternativa era mas directa conceptualmente pero implicaba una
  query extra por cada llamada a E38/E41 y N queries en E39.
  saldoHoras ya acumula la diferencia exacta del proceso nocturno,
  por lo que despejar trabajadas es equivalente y sin coste adicional.

Decision T3-B -- SaldoResponse con clases estaticas internas
  Los tres desgloses (vacaciones, asuntos propios, horas) se modelan
  como clases estaticas internas de SaldoResponse en lugar de ficheros
  separados. Solo se usan dentro de SaldoResponse, no tienen sentido
  fuera de ese contexto, y no se reutilizan en ningun otro DTO.
  Alternativa descartada: 3 ficheros separados en dto.response.
  No aporta ninguna ventaja y fragmenta innecesariamente el codigo.

Decision T3-C -- Patron findOrCreate en E40
  E40 recalcular usa orElseGet(() -> crearSaldoInicial()) en lugar
  de lanzar 404 si no existe el registro. Razon: E40 es una operacion
  de escritura que debe funcionar incluso antes de que el proceso
  nocturno haya creado el registro del ano. Un ADMIN debe poder
  forzar el recalculo en cualquier momento, incluido el primer dia
  del ano antes de que ProcesoDiario haya actuado.
  Alternativa descartada: devolver 404 si no existe el registro.
  Obligaria al ADMIN a esperar al proceso nocturno o a insertar
  datos manualmente en la BD antes de poder recalcular.

Decision T3-D -- Idempotencia en E40 por reinicio de contadores
  Antes de iterar los fichajes E40 reinicia todos los contadores
  a 0 (diasTrabajados, diasVacacionesConsumidos, saldoHoras, etc.).
  Esto garantiza que ejecutar E40 varias veces produce siempre el
  mismo resultado (idempotencia). Alternativa descartada: acumular
  sobre los valores existentes. Si se acumulara sobre valores previos,
  la segunda ejecucion doblaria todos los contadores.

Decision T3-E -- obtenerMiSaldo recibe String username (no Long empleadoId)
  El controller extrae el username con authentication.getName() y lo
  pasa al service. El service resuelve el empleado con
  findByUsuarioUsername(username). Se descarto pasar el empleadoId
  directamente desde el claim del JWT porque authentication.getDetails()
  devuelve WebAuthenticationDetails (IP y sessionId), no los claims.
  Acceder a claims del JWT desde el controller requeriria implementacion
  adicional en JwtAuthFilter para exponerlos en el Authentication.
  El patron username -> findByUsuarioUsername ya esta establecido en
  el proyecto (E34, E37) y es mas simple y coherente.

Decision T3-F -- @Transactional solo en E40
  Solo E40 (recalcular) tiene @Transactional porque es la unica
  operacion de escritura del grupo. E38, E39 y E41 son consultas
  de solo lectura: anotarlas con @Transactional(readOnly=true) seria
  correcto pero superfluo para el alcance del proyecto. Spring gestiona
  las transacciones de lectura automaticamente en JPA sin necesidad
  de anotacion explicita.
