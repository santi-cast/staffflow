1. CONTEXTO
====================================================

La Tarea 2 del Bloque 6 implementa el Grupo 8 de la API REST:
control de presencia en tiempo real. Es el unico grupo del sistema
que no crea ni modifica ningun dato: todos sus endpoints son GET
de solo lectura que calculan el estado de cada empleado aggregando
informacion de tres tablas (fichajes, pausas, planificacion_ausencias).

Antes de escribir codigo se revisaron los contratos de E35-E37
en StaffFlow_Endpoints_v3.docx y se leyeron las entidades Fichaje,
Pausa, Empleado y los repositorios FichajeRepository, PausaRepository,
EmpleadoRepository para conocer los nombres exactos de campos y
metodos existentes antes de anadir nada nuevo.

Decision de diseno previa verificada en el docx:
  PresenciaController esta separado de FichajeController por
  responsabilidad unica. Los fichajes son el registro historico
  inmutable (RD-ley 8/2019). La presencia es una vista calculada
  en tiempo real. Mezclarlos confundiria las responsabilidades.
  El docx lo llama "patron Query Object" (vista calculada,
  sin escritura, sin estado propio).

Refactoring de DTOs realizado en esta tarea:
  El ParteDiarioResponse.java existente (esqueleto del Bloque 1)
  representaba un empleado individual. El contrato de E35 exige
  un objeto raiz con contadores globales mas un array detalle[].
  Se tomo la decision de renombrar el DTO existente a
  DetallePresenciaResponse y crear un ParteDiarioResponse nuevo
  con la estructura correcta. Decision de nomenclatura profesional:
  los nombres deben ser autodescriptivos sin necesidad de contexto.

E35 -- GET /api/v1/presencia/parte-diario
RF-30 (parte diario completo)
----------------------------------------------------

Roles: ADMIN, ENCARGADO
Autenticacion: JWT requerido

Query params:
  ?fecha=YYYY-MM-DD   opcional -- por defecto hoy (LocalDate.now())

Respuesta 200:
  {
    "fecha": "2026-03-13",
    "totalEmpleados": n,
    "fichados": n,
    "enPausa": n,
    "ausencias": n,
    "sinJustificar": n,
    "detalle": [
      { empleadoId, nombre, apellido1, apellido2, estado,
        horaEntrada, horaSalida, pausaActiva, fichajeTipo }
    ]
  }

Logica del service:
  1. 4 queries planas en paralelo conceptual (sin N+1):
       findByActivo(true)                  --> todos los empleados activos
       findByFechaWithEmpleado(fecha)      --> fichajes del dia con empleado
       findPausasActivasByFecha(fecha)     --> pausas activas del dia
       findByFechaAndProcesadoFalse(fecha) --> ausencias planificadas del dia
  2. Construye Maps y Sets en memoria para clasificacion O(n).
  3. Detecta si hay festivo global (ausencia con empleado=null).
  4. Clasifica cada empleado con clasificarEmpleado() (metodo privado).
  5. Acumula contadores globales en el switch por estado.
  6. Ordena detalle[] por apellido1, apellido2, nombre.
  7. Devuelve ParteDiarioResponse con contadores + detalle.

Nota sobre el contador fichados:
  Incluye EN_PAUSA porque un empleado en pausa tambien cuenta
  como fichado. El contador enPausa es subconjunto de fichados.

Codigos HTTP:
  200 OK   parte diario con contadores y detalle
  403      rol insuficiente

E36 -- GET /api/v1/presencia/sin-justificar
RF-31 (empleados sin justificacion)
----------------------------------------------------

Roles: ADMIN, ENCARGADO
Autenticacion: JWT requerido

Query params:
  ?fecha=YYYY-MM-DD   opcional -- por defecto hoy

Respuesta 200:
  [ { empleadoId, nombre, apellido1, apellido2 }, ... ]
  Lista vacia si todos tienen registro o si hay festivo global.

Logica del service:
  1. Mismas 4 queries que E35 (mismos datos, misma carga).
  2. Si hay festivo global --> devuelve List.of() inmediatamente.
     Razon: todos los empleados tienen AUSENCIA_PLANIFICADA ese dia,
     nadie esta sin justificar aunque el proceso nocturno aun no
     haya generado los fichajes.
  3. Filtra empleados que no tienen fichaje, no estan en pausa
     y no tienen ausencia planificada individual.
  4. Devuelve List<SinJustificarResponse> ordenada por apellido1.

Codigos HTTP:
  200 OK   lista de empleados sin justificar (puede ser vacia)
  403      rol insuficiente

E37 -- GET /api/v1/presencia/parte-diario/me
RF-54 (estado propio del empleado autenticado)
----------------------------------------------------

Roles: EMPLEADO
Autenticacion: JWT requerido

Query params:
  ?fecha=YYYY-MM-DD   opcional -- por defecto hoy

Respuesta 200:
  { empleadoId, nombre, apellido1, apellido2, estado,
    horaEntrada, horaSalida, pausaActiva, fichajeTipo }

Logica del service:
  1. authentication.getName() --> username --> findByUsuarioUsername()
     --> empleado. Si no tiene perfil de empleado: 404.
  2. Carga solo los datos del empleado concreto (no carga todos):
       findByEmpleadoIdAndFecha(id, fecha)              --> su fichaje
       findByEmpleadoIdAndFechaAndHoraFinIsNull(id, f)  --> pausa activa
       findByFechaAndProcesadoFalse(fecha)              --> ausencias del dia
  3. Reutiliza clasificarEmpleado() para garantizar que el estado
     calculado es identico al que ve el encargado en E35.
  4. Devuelve DetallePresenciaResponse con su estado.

Codigos HTTP:
  200 OK   estado propio del dia
  403      acceso denegado (rol incorrecto)
  404      usuario sin perfil de empleado

4. REPOSITORY -- METODOS CUSTOM ANADIDOS
====================================================

FichajeRepository -- 1 metodo nuevo:

  findByFechaWithEmpleado(LocalDate fecha)
    @Query JPQL con JOIN FETCH f.empleado.
    Devuelve todos los fichajes de una fecha con el empleado cargado.
    Se usa @Query explicita porque el metodo convencional findByFecha()
    no haria el JOIN FETCH y provocaria N+1 al acceder a empleado.id
    en el service. Con JOIN FETCH el empleado llega cargado en la
    misma query, sin lazy loading adicional.

PausaRepository -- 1 metodo nuevo:

  findPausasActivasByFecha(LocalDate fecha)
    @Query JPQL con JOIN FETCH p.empleado y filtro horaFin IS NULL.
    Devuelve solo las pausas activas (en curso) de una fecha.
    Una pausa activa es aquella con horaFin = null: el empleado
    esta en pausa en este momento. El service construye un Set de
    empleadoIds con pausa activa para clasificacion en memoria.

PlanificacionAusenciaRepository -- 1 metodo nuevo:

  findByFechaAndProcesadoFalse(LocalDate fecha)
    @Query JPQL con LEFT JOIN FETCH a.empleado.
    LEFT JOIN porque empleado puede ser null (festivos globales, RF-26).
    Devuelve ausencias planificadas pendientes de proceso para una fecha.
    Se usa para dos cosas: detectar festivos globales (empleado=null)
    y detectar ausencias individuales (empleado!=null) de cada empleado.

====================================================
5. LOGICA DE CLASIFICACION -- EstadoPresencia
====================================================

El metodo privado clasificarEmpleado() aplica la logica de prioridad
definida en el Javadoc de EstadoPresencia.java (D-012).
Se implementa como metodo privado compartido entre obtenerParteDiario()
y obtenerMiPresencia() para garantizar que la clasificacion es identica
en E35 y E37. Alternativa descartada: duplicar la logica en cada metodo,
con mayor riesgo de inconsistencia futura.

Orden de prioridad:

  1. EN_PAUSA
     Condicion: empleado tiene pausa con horaFin = null hoy.
     Tiene prioridad sobre el estado del fichaje porque aunque el
     empleado tenga fichaje abierto, en este momento esta en pausa.

  2. JORNADA_COMPLETADA
     Condicion: fichaje con horaEntrada != null Y horaSalida != null.
     Jornada cerrada: el empleado ha terminado su dia.

  3. JORNADA_INICIADA
     Condicion: fichaje con horaEntrada != null Y horaSalida = null
     (y sin pausa activa, ya cubierta por prioridad 1).
     Jornada en curso: el empleado esta trabajando ahora mismo.

  4. AUSENCIA_REGISTRADA
     Condicion: fichaje con horaEntrada = null (tipo != NORMAL).
     El proceso diario genero un fichaje de ausencia (VACACIONES,
     BAJA_MEDICA, etc.) sin hora de entrada porque no es jornada
     presencial. Es una ausencia ya materializada en fichajes.

  5. AUSENCIA_PLANIFICADA
     Condicion: no tiene fichaje, pero tiene ausencia en
     planificacion_ausencias con procesado=false, O existe un
     festivo global para ese dia.
     El proceso nocturno aun no ha convertido la ausencia en fichaje.

  6. SIN_JUSTIFICAR
     Condicion: ninguno de los anteriores.
     El empleado no tiene ningun registro para ese dia.
     Requiere atencion del encargado (RF-31).

Caso especial -- festivo global:
  Si existe un registro en planificacion_ausencias con empleado=null
  para la fecha, todos los empleados sin fichaje tienen estado
  AUSENCIA_PLANIFICADA en lugar de SIN_JUSTIFICAR. Esto evita que
  un festivo nacional marque a todos como sin justificar cuando el
  proceso nocturno aun no ha actuado (normalmente hasta las 00:01).

6. DECISIONES TECNICAS DE ESTA TAREA
====================================================

Decision T2-A -- Patron antiN+1: 4 queries planas + clasificacion en memoria
  En lugar de cargar cada empleado y lanzar queries individuales para
  su fichaje, pausa y ausencia (N*3 queries), el service carga las 4
  colecciones completas del dia en 4 queries fijas y clasifica en memoria
  con Maps y Sets. Para 50 empleados: 4 queries vs 150 queries.
  Alternativa descartada: cargar datos por empleado en bucle -- correcto
  funcionalmente pero ineficiente y no escalable.

Decision T2-B -- DetallePresenciaResponse compartido entre E35 y E37
  E37 /me devuelve exactamente la misma estructura que cada fila del
  array detalle[] de E35. Se usa el mismo DTO para ambos endpoints.
  Alternativa descartada: crear un MiPresenciaResponse especifico para
  E37 con un subconjunto de campos -- anade complejidad sin beneficio
  real porque el empleado ve exactamente lo mismo que el encargado
  ve sobre el.

Decision T2-C -- clasificarEmpleado() como metodo privado compartido
  La logica de clasificacion se extrae a un metodo privado reutilizado
  por obtenerParteDiario() y obtenerMiPresencia(). Garantiza que el
  estado que ve el empleado en E37 es identico al que ve el encargado
  en E35 para ese mismo empleado. Coherencia garantizada por diseno.

Decision T2-D -- Renombrado ParteDiarioResponse --> DetallePresenciaResponse
  El DTO existente en el Bloque 1 se llamaba ParteDiarioResponse pero
  representaba un empleado individual, no el parte completo. Se renombra
  a DetallePresenciaResponse (mas preciso) y se crea un nuevo
  ParteDiarioResponse que representa el objeto raiz con contadores.
  Criterio: los nombres deben ser autodescriptivos sin necesidad de
  leer el codigo para entender que representan.

Decision T2-E -- @RequiredArgsConstructor en lugar de constructor manual
  El service usa @RequiredArgsConstructor de Lombok para mantener el
  estilo del proyecto (todos los services y controllers del proyecto
  usan esta anotacion). Lombok genera el constructor con los 4 campos
  final automaticamente. Funcionalmente equivalente al constructor manual.

Decision T2-F -- festivo global devuelve List.of() en E36
  Si existe un festivo global para la fecha, obtenerSinJustificar()
  devuelve una lista vacia inmediatamente sin iterar los empleados.
  Razon: con festivo global todos tienen AUSENCIA_PLANIFICADA ese dia,
  la lista de sin justificar es necesariamente vacia. Cortocircuito
  que evita procesamiento innecesario.

7.1 METODOLOGIA
----------------------------------------------------

Herramienta utilizada: PowerShell con Invoke-WebRequest -UseBasicParsing.

Invoke-WebRequest es el cmdlet de PowerShell equivalente a curl en Linux.
Permite lanzar peticiones HTTP directamente desde la consola de Windows
sin instalar herramientas externas. El parametro -UseBasicParsing evita
que PowerShell intente parsear la respuesta como HTML, lo que en algunas
versiones de Windows causa errores si no hay Internet Explorer instalado.
Con -UseBasicParsing la respuesta se trata como texto plano, que es lo
que necesitamos para leer el JSON del backend.

El resultado de Invoke-WebRequest se guarda en una variable (por ejemplo
$e35, $e36, $e37). Esta variable es un objeto que tiene dos propiedades
relevantes:
  .StatusCode  -- el codigo HTTP de la respuesta (200, 403, 404, etc.)
  .Content     -- el cuerpo de la respuesta como texto, en este caso JSON.

Guardar la respuesta en una variable en lugar de imprimirla directamente
permite inspeccionarla por partes: primero el status, luego el contenido,
sin repetir la peticion. Es el mismo principio que guardar la respuesta
de fetch() en una variable en JavaScript antes de llamar a .json().

El token JWT se obtiene de la misma forma: se lanza el login, se guarda
el objeto de respuesta en $r (para admin) o $r2 (para empleado), se
parsea el JSON con ConvertFrom-Json y se extrae el campo .token. A partir
de ahi $token y $tokenEmp se reutilizan en todas las peticiones
autenticadas del dia sin necesidad de volver a hacer login.

Razon de usar PowerShell en lugar de Swagger UI:
  Swagger UI requiere copiar y pegar el token manualmente en cada
  peticion. PowerShell permite encadenar peticiones reutilizando
  la variable $token automaticamente. Para verificaciones con varios
  endpoints seguidos es mas rapido y menos propenso a errores de
  copia de token.

Estado de la BD al iniciar la verificacion:

    donde se anticipaba totalEmpleados en SIN_JUSTIFICAR. El comportamiento

7.2 CONFIGURACION INICIAL: TOKEN JWT
----------------------------------------------------

Token admin:
  $r = Invoke-WebRequest
    -Uri "http://localhost:8080/api/v1/auth/login"
    -Method POST
    -ContentType "application/json"
    -Body '{"username":"admin","password":"admin1234"}'
    -UseBasicParsing
  $token = ($r.Content | ConvertFrom-Json).token

  Que hace cada linea:
    Invoke-WebRequest lanza un POST al endpoint de login con el body
    JSON de credenciales. La respuesta completa (cabeceras + body)
    queda en $r. La segunda linea extrae el campo "token" del JSON:
    $r.Content es el body como string, ConvertFrom-Json lo convierte
    en objeto PowerShell, y .token accede al campo. El resultado es
    la cadena JWT que se usara en el header Authorization de las
    siguientes peticiones.

Token empleado (emp01):
  $r2 = Invoke-WebRequest
    -Uri "http://localhost:8080/api/v1/auth/login"
    -Method POST
    -ContentType "application/json"
    -Body '{"username":"emp01","password":"admin1234"}'
    -UseBasicParsing
  $tokenEmp = ($r2.Content | ConvertFrom-Json).token

  Se usa $r2 y $tokenEmp para no sobreescribir las variables del admin.
  A partir de aqui $token es el JWT de admin y $tokenEmp el de empleado.

Verificacion de tokens:
  Write-Host "Admin token OK:" ($token.Length -gt 0)   --> True
  Write-Host "Emp token OK:" ($tokenEmp.Length -gt 0)  --> True

  Length -gt 0 comprueba que la variable no esta vacia, es decir,
  que el login fue exitoso y se recibio un token. True en ambos casos.
