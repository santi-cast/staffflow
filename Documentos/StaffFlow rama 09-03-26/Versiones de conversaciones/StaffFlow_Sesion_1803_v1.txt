====================================================
2. DECISIONES TOMADAS AL INICIO
====================================================

DECISION 1 -- Reutilizacion logica recalculo en Tarea C:
  Opcion elegida: A (preferida).
  Razonamiento explicado a Santiago:
    recalcular() (E40) hace dos cosas: calcula y construye DTO.
    ProcesoCierreDiario no necesita el DTO (proceso interno).
    Opcion A extrae la logica a recalcularParaProceso() sin DTO.
    recalcular() llama internamente a recalcularParaProceso() y
    luego construye el SaldoResponse para devolverlo al cliente.
    Mas limpio, sin trabajo innecesario cada noche.

DECISION 2 -- Como pasa el rol el controller al service:
  Opcion elegida: B.
  Razonamiento: consistente con patron D-017 del proyecto.
  Implementacion: controller pasa username. Service llama a
    usuarioRepository.findByUsername(username) y compara
    usuario.getRol() == Rol.ENCARGADO.
  Nota: en E22 y E23 (FichajeService) el usuario ya se resuelve
  por otros motivos (auditoria), asi que la query extra de D-017
  no supone un coste adicional real. En E27/E28 y E30/E31 si hay
  una query adicional por peticion, aceptable para PYME.

ProcesoCierreDiario.java -- Implementacion completa
----------------------------------------------------

Paquete: com.staffflow.service.scheduled
Renombrado desde ProcesoDiarioService (paquete service).
Anotaciones: @Slf4j @Service @RequiredArgsConstructor.
Cron: "0 55 23 * * *".
Dependencias inyectadas: ausenciaRepository, fichajeRepository,
  empleadoRepository, usuarioRepository, saldoService.

ejecutar() es @Scheduled y @Transactional.
Patron: cargar usuarioSistema y empleadosActivos una vez al inicio
  para reutilizar en las tres tareas sin queries repetidas.

TAREA A:
  Para cada empleado activo: findByEmpleadoIdAndFecha().isPresent().
  Si no existe fichaje: crear AUSENCIA_INJUSTIFICADA con campos
  horaEntrada=null, horaSalida=null, jornadaEfectivaMinutos=0,
  totalPausasMinutos=0, usuario=terminal_service.

TAREA B:
  findPendientesByFechaLessThanEqual(hoy.plusDays(1)).
  Para cada planificacion:
    Si empleado==null (festivo global): iterar empleados activos,
      crearFichajeAusenciaSiNoExiste() para cada uno.
    Si empleado!=null: crearFichajeAusenciaSiNoExiste() solo para ese.
  Marcar procesado=true y save().
  Mapeo: TipoFichaje.valueOf(tipoAusencia.name()) -- directo por nombre.

TAREA C:
  Para cada empleado activo: saldoService.recalcularParaProceso().

crearFichajeAusenciaSiNoExiste():
  Metodo privado. Verifica existencia antes de crear.
  Evita duplicados si Tarea A ya creo AUSENCIA_INJUSTIFICADA y
  luego Tarea B intenta crear fichaje para festivo global del mismo dia.

mapearTipoAusenciaATipoFichaje():
  Metodo privado. TipoFichaje.valueOf(tipoAusencia.name()).
  Falla con IllegalArgumentException si hay discrepancia de enums
  en lugar de producir datos incorrectos silenciosamente.

Log completo: inicio/fin de cada tarea con contadores.

6. NOTA SOBRE MYSQL PROD
====================================================

terminal_service debe existir en el script de inicializacion de
MySQL prod con el mismo username "terminal_service".
ProcesoCierreDiario lo busca por username (no por id) para no
depender del autoincremental de BD que puede diferir entre dev y prod.
Si terminal_service no existe en prod, el proceso falla a las 23:55
con IllegalStateException y ningun fichaje automatico se crea ese dia.
