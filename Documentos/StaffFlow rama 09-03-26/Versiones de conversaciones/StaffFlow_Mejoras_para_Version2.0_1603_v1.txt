====================================================
STAFFFLOW -- MEJORAS PARA VERSION 2.0
====================================================

Este fichero recoge las mejoras identificadas durante el desarrollo
de StaffFlow v1.0 que quedan fuera del alcance del TFG pero que
deberian implementarse en una version posterior.

No son errores ni desviaciones del diseno: son limitaciones
conocidas y asumidas de v1.0, documentadas para trazabilidad
y como base de un roadmap futuro.

Ultima actualizacion: 16/03/2026 (sesion 16)

Convenio de nombres:
  StaffFlow_Mejoras_para_Version2.0_[DDMM]_v[N].txt
  La version reinicia a v1 cada dia nuevo.

====================================================
MEJORAS IDENTIFICADAS
====================================================

----------------------------------------------------
MEJORA M-001
----------------------------------------------------
AREA: Seguridad -- Autenticacion JWT
DETECTADA: 14/03/2026 (D-024, sesion memoria)
QUE FALTA EN V1.0:
  JWT con duracion fija de 12 horas. Si el token expira durante
  la jornada el usuario debe hacer login de nuevo. No hay mecanismo
  de renovacion transparente.
MEJORA PROPUESTA PARA V2.0:
  Implementar refresh token: un token de larga duracion (7 dias)
  que permite renovar el JWT de corta duracion (1-2 horas) de forma
  transparente sin que el usuario tenga que hacer login de nuevo.
  Requiere nuevo endpoint POST /api/v1/auth/refresh y campo
  refresh_token en la tabla usuarios.
BENEFICIO:
  Combina seguridad (tokens JWT cortos) con buena usabilidad
  (sin reautenticaciones visibles para el usuario).

----------------------------------------------------
MEJORA M-002
----------------------------------------------------
AREA: Operativa diaria -- Cierre de dia
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  El cierre del dia es automatico a las 23:55 mediante
  ProcesoCierreDiario. El ENCARGADO no tiene forma de adelantar
  el cierre cuando considera que el dia ya esta completo, por
  ejemplo a las 18:00 cuando todos los empleados han salido.
MEJORA PROPUESTA PARA V2.0:
  Nuevo endpoint POST /api/v1/presencia/cierre-dia que el ENCARGADO
  puede invocar manualmente. Antes de ejecutar el cierre, el sistema
  verifica que todos los empleados activos tienen fichaje completo
  (usando la logica de E25 y E36). Si hay empleados pendientes,
  avisa al ENCARGADO antes de proceder. Si todo esta correcto,
  ejecuta las tres tareas de ProcesoCierreDiario inmediatamente.
BENEFICIO:
  El ENCARGADO tiene control total sobre el momento del cierre
  sin depender del proceso automatico de las 23:55.

----------------------------------------------------
MEJORA M-003
----------------------------------------------------
AREA: Seguridad -- Control de acciones sensibles
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  Ciertas acciones que puede ejecutar el ENCARGADO tienen impacto
  directo en saldos y registros de otros empleados. En v1.0 no hay
  ningun mecanismo de supervision del ADMIN para estas acciones.
  Por ejemplo, la modificacion de un fichaje retroactivo (E23)
  dispara automaticamente un recalculo de saldos sin que el ADMIN
  lo autorice explicitamente.
MEJORA PROPUESTA PARA V2.0:
  Mecanismo de autorizacion tipo sudo: para acciones sensibles
  definidas (modificacion de fichajes retroactivos, recalculo
  de saldos, etc.) el sistema solicita la contrasena del ADMIN
  antes de ejecutar la accion. El ENCARGADO no puede completar
  la accion sin la intervencion explicita del ADMIN.
  Requiere nuevo endpoint POST /api/v1/auth/sudo y logica
  de verificacion en los endpoints afectados.
BENEFICIO:
  El ADMIN mantiene supervision real sobre las acciones sensibles
  del ENCARGADO. Reduce el riesgo de manipulacion de saldos
  o registros por error o de forma malintencionada.

----------------------------------------------------
MEJORA M-004
----------------------------------------------------
AREA: Saldos anuales -- Control de acceso
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  En v1.0 solo el ADMIN puede recalcular saldos manualmente
  mediante E40. El ENCARGADO no tiene ningun endpoint para
  solicitar un recalculo, aunque puede provocarlo indirectamente
  modificando un fichaje con E23. Esto no era el diseno previsto:
  idealmente el ENCARGADO no deberia poder afectar a los saldos
  en absoluto. Se asume el riesgo en v1.0 como limitacion conocida.
MEJORA PROPUESTA PARA V2.0:
  Nuevo endpoint POST /api/v1/saldos/{empleadoId}/recalcular
  accesible por ENCARGADO, protegido por el mecanismo sudo de M-003.
  El ENCARGADO solicita el recalculo, el sistema pide autorizacion
  del ADMIN, y solo entonces ejecuta el recalculo.
BENEFICIO:
  El ENCARGADO tiene un canal oficial para solicitar recalculos
  sin acceder indirectamente a los saldos via E23. El ADMIN
  mantiene el control real sobre cuando y como se recalculan
  los saldos.

----------------------------------------------------
MEJORA M-005
----------------------------------------------------
AREA: Modelo de datos -- Trazabilidad de saldos
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  Cuando ProcesoCierreDiario o E40 recalculan un saldo, el valor
  anterior se sobreescribe sin dejar rastro. Si hay una discrepancia
  en el saldo de un empleado no hay forma de saber cuando ocurrio
  el cambio, que lo provoco ni cual era el valor previo.
MEJORA PROPUESTA PARA V2.0:
  Nueva tabla historial_saldos que registra cada recalculo con:
    - empleado_id
    - anio
    - fecha_recalculo
    - motivo (ProcesoCierreDiario / E40 manual / E23 retroactivo)
    - usuario_id (quien lo disparo, o sistema si fue automatico)
    - snapshot del saldo anterior (JSON o campos individuales)
    - snapshot del saldo nuevo
  Requiere DDL v6 y nueva entidad JPA HistorialSaldo.
BENEFICIO:
  Trazabilidad completa de la evolucion de los saldos. Permite
  al ADMIN auditar cualquier cambio y detectar manipulaciones
  o errores del proceso automatico.

====================================================
LIMITACIONES CONOCIDAS DE V1.0
====================================================

Las siguientes limitaciones son consecuencia directa de las
mejoras pendientes. Se documentan aqui para que puedan
mencionarse en la memoria final del TFG como trabajo futuro.

L-001: El ENCARGADO puede afectar indirectamente a los saldos
  anuales modificando fichajes con E23. No era el diseno previsto.
  Riesgo asumido en v1.0. Solucion en M-003 y M-004.

L-002: No hay trazabilidad de los cambios en saldos anuales.
  Si un saldo esta mal no hay forma de saber cuando ni por que
  cambio. Solucion en M-005.

L-003: El ENCARGADO no puede adelantar el cierre del dia.
  Debe esperar a las 23:55 para que ProcesoCierreDiario actue.
  Solucion en M-002.

L-004: El JWT de 12h no se puede renovar de forma transparente.
  Si expira durante la jornada el usuario debe hacer login de nuevo.
  Solucion en M-001.
