====================================================
3. DECISIONES CLAVE DE LA SESION
====================================================

----------------------------------------------------
DECISION A -- iText 7 en pom.xml
----------------------------------------------------

iText 7 NO se anade al pom.xml en esta sesion.
Razon: E42 y E43 solo generan HTML como String. No se necesita
ninguna dependencia adicional para eso.
iText 7 se anadira en la proxima sesion antes de implementar
E44-E47 (informes PDF). Si se anade ahora habria que verificar
que el arranque sigue limpio, y no hay nada que probar todavia.

----------------------------------------------------
DECISION B -- No usar libreria de plantillas HTML
----------------------------------------------------

El HTML se genera con StringBuilder en el service.
Alternativas descartadas: Thymeleaf, Freemarker.
Razon: anadir una dependencia de plantillas para un unico informe
es sobredimensionado. El HTML es autocontenido con CSS inline.
El metodo htmlCabecera() genera el bloque <style> completo.

----------------------------------------------------
DECISION C -- Clases internas privadas en InformeService
----------------------------------------------------

Se usaron tres clases internas privadas para el modelo de trabajo
interno del service: DiaInforme, PausaInforme, ResumenInforme.
Estas clases no se exponen fuera del service.
Razon: evitar crear DTOs de response para el informe cuando la
unica salida es un Map<String, Object> (JSON) o un String (HTML).
Los DTOs de response se reservan para la API publica.

----------------------------------------------------
DECISION D -- Metodo nuevo en PausaRepository
----------------------------------------------------

Se anadio findByEmpleadoIdAndFechaBetween() al final de
PausaRepository en lugar de reutilizar findByFiltros().
Razon: findByFiltros() hace JOIN FETCH con el empleado, lo cual
es innecesario en el contexto del informe donde el empleado ya
esta resuelto. Un metodo Spring Data convencional es mas limpio
y eficiente para este caso de uso especifico.

----------------------------------------------------
DECISION E -- Desviacion D-027: parametro ?tipo=
----------------------------------------------------

Durante el diseno del informe se identifico la necesidad de
poder filtrar el detalle por tipo de jornada. El diseno original
de E42 y E43 no incluia este parametro.
Se documento como D-027 en lugar de modificar
StaffFlow_Endpoints_v3.docx (snapshot cerrado).
El comportamiento sin ?tipo= es identico al diseno original:
sin filtro se devuelven todos los dias del periodo.

----------------------------------------------------
DECISION F -- DIA_LIBRE y SIN_REGISTRO como tipos virtuales
----------------------------------------------------

DIA_LIBRE y SIN_REGISTRO no existen en el enum TipoFichaje.
Son tipos virtuales del informe que representan dias sin fichaje.
DIA_LIBRE: sabado o domingo sin fichaje en BD.
SIN_REGISTRO: dia entre semana sin fichaje en BD.
Ambos aceptados como valores del parametro ?tipo= de D-027.
Ambos generan filas con guion en todas las columnas.
SIN_REGISTRO es util para periodos futuros donde aun no hay
fichajes registrados.
