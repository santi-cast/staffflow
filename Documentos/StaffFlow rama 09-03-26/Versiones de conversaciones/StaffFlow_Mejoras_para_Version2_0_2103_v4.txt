====================================================
STAFFFLOW -- MEJORAS PARA VERSION 2.0
====================================================

Este fichero recoge las mejoras identificadas durante el desarrollo
de StaffFlow v1.0 que quedan fuera del alcance del TFG pero que
deberian implementarse en una version posterior.

No son errores ni desviaciones del diseno: son limitaciones
conocidas y asumidas de v1.0, documentadas para trazabilidad
y como base de un roadmap futuro.

Ultima actualizacion: 21/03/2026 (sesion 20)

Convenio de nombres:
  StaffFlow_Mejoras_para_Version2.0_[DDMM]_v[N].txt
  La version reinicia a v1 cada dia nuevo.

====================================================
MEJORAS IDENTIFICADAS
====================================================

----------------------------------------------------
MEJORA M-001
----------------------------------------------------
AREA: Seguridad -- Autenticacion JWT
DETECTADA: 14/03/2026 (D-024, sesion memoria)
QUE FALTA EN V1.0:
  JWT con duracion fija de 12 horas. Si el token expira durante
  la jornada el usuario debe hacer login de nuevo. No hay mecanismo
  de renovacion transparente.
MEJORA PROPUESTA PARA V2.0:
  Implementar refresh token: un token de larga duracion (7 dias)
  que permite renovar el JWT de corta duracion (1-2 horas) de forma
  transparente sin que el usuario tenga que hacer login de nuevo.
  Requiere nuevo endpoint POST /api/v1/auth/refresh y campo
  refresh_token en la tabla usuarios.
BENEFICIO:
  Combina seguridad (tokens JWT cortos) con buena usabilidad
  (sin reautenticaciones visibles para el usuario).

----------------------------------------------------
MEJORA M-002
----------------------------------------------------
AREA: Operativa diaria -- Cierre de dia
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  En v1.0 el cierre del dia lo ejecuta ProcesoCierreDiario
  automaticamente a las 23:55. El ENCARGADO no tiene forma de
  indicar al sistema que el dia ya esta cerrado antes de esa hora.
  La restriccion de que el ENCARGADO solo puede modificar registros
  del dia actual se implementa validando que la fecha sea hoy en
  E22, E23, E27, E28, E30 y E31. Esto significa que el ENCARGADO
  puede seguir modificando registros del dia actual hasta las 23:55
  aunque ya haya considerado el dia cerrado.
MEJORA PROPUESTA PARA V2.0:
  Nuevo endpoint POST /api/v1/presencia/cierre-dia accesible por
  ENCARGADO. Antes de ejecutar el cierre, el sistema verifica con
  la logica de E25 y E36 que todos los empleados activos tienen
  fichaje completo. Si hay empleados pendientes, avisa al ENCARGADO
  antes de proceder. Si todo esta correcto:
    1. Ejecuta las tres tareas de ProcesoCierreDiario inmediatamente.
    2. Persiste el estado de cierre del dia en BD (nuevo campo o tabla).
    3. A partir de ese momento bloquea E22, E23, E27, E28, E30 y E31
       para el ENCARGADO aunque no sean las 23:55, devolviendo 409
       con mensaje "El dia ya ha sido cerrado".
  ADMIN no queda bloqueado por el cierre: puede seguir modificando
  cualquier dia sin restriccion.
BENEFICIO:
  El ENCARGADO tiene control total sobre el momento del cierre.
  El bloqueo es inmediato al pulsar el boton, no dependiente del
  paso del tiempo hasta las 23:55. Reduce el riesgo de modificaciones
  accidentales despues de que el ENCARGADO considera el dia cerrado.

----------------------------------------------------
MEJORA M-003
----------------------------------------------------
AREA: Seguridad -- Control de acciones sensibles del ENCARGADO
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  En v1.0 el ENCARGADO esta limitado a modificar solo registros
  del dia actual mediante validacion de fecha en E22, E23, E27,
  E28, E30 y E31. Las correcciones retroactivas de dias pasados
  son exclusivas del ADMIN. Sin embargo, no hay ningun mecanismo
  de supervision explicita del ADMIN para las acciones que el
  ENCARGADO si puede ejecutar durante el dia actual.
MEJORA PROPUESTA PARA V2.0:
  Mecanismo de autorizacion tipo sudo para acciones sensibles:
  cuando el ENCARGADO intenta ejecutar ciertas acciones definidas
  como sensibles (por ejemplo, modificar el tipo de fichaje de un
  empleado, cambiar una ausencia planificada a ultimo momento, etc.)
  el sistema solicita la contrasena del ADMIN antes de ejecutar
  la accion. El ENCARGADO no puede completar la accion sin la
  intervencion explicita del ADMIN.
  Requiere nuevo endpoint POST /api/v1/auth/sudo y logica de
  verificacion en los endpoints afectados.
BENEFICIO:
  El ADMIN mantiene supervision sobre las acciones mas sensibles
  del dia actual, no solo sobre las correcciones retroactivas.

----------------------------------------------------
MEJORA M-004
----------------------------------------------------
AREA: Saldos anuales -- Acceso del ENCARGADO
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  En v1.0 solo el ADMIN puede recalcular saldos manualmente
  mediante E40. El ENCARGADO no tiene ningun canal oficial para
  solicitar un recalculo si detecta una discrepancia. Los saldos
  se actualizan automaticamente cada noche via ProcesoCierreDiario
  Tarea C, y manualmente via E40 (solo ADMIN).
MEJORA PROPUESTA PARA V2.0:
  Nuevo endpoint POST /api/v1/saldos/{empleadoId}/recalcular/solicitar
  accesible por ENCARGADO, protegido por el mecanismo sudo de M-003.
  El ENCARGADO solicita el recalculo, el sistema pide autorizacion
  del ADMIN mediante sudo, y solo entonces ejecuta el recalculo.
BENEFICIO:
  El ENCARGADO tiene un canal oficial para solicitar recalculos
  cuando detecta discrepancias. El ADMIN mantiene el control real
  sobre cuando y como se recalculan los saldos.

----------------------------------------------------
MEJORA M-005
----------------------------------------------------
AREA: Modelo de datos -- Trazabilidad de saldos
DETECTADA: 16/03/2026 (sesion 16)
QUE FALTA EN V1.0:
  Cuando ProcesoCierreDiario o E40 recalculan un saldo, el valor
  anterior se sobreescribe sin dejar rastro. Si hay una discrepancia
  en el saldo de un empleado no hay forma de saber cuando ocurrio
  el cambio, que lo provoco ni cual era el valor previo.
MEJORA PROPUESTA PARA V2.0:
  Nueva tabla historial_saldos que registra cada recalculo con:
    - empleado_id
    - anio
    - fecha_recalculo
    - motivo (ProcesoCierreDiario / E40 manual / solicitud ENCARGADO)
    - usuario_id (quien lo disparo, o sistema si fue automatico)
    - snapshot del saldo anterior (JSON o campos individuales)
    - snapshot del saldo nuevo
  Requiere DDL v6 y nueva entidad JPA HistorialSaldo.
BENEFICIO:
  Trazabilidad completa de la evolucion de los saldos. Permite
  al ADMIN auditar cualquier cambio y detectar errores del proceso
  automatico.

MEJORA M-006
----------------------------------------------------
AREA: Modelo de datos -- Jornada mixta
DETECTADA: 21/03/2026 (sesion 18)
QUE FALTA EN V1.0:
  En v1.0 un dia solo puede tener un tipo de fichaje. Si un
  empleado tiene permiso retribuido parcial (por ejemplo, va a
  votar por la manana y trabaja el resto del dia), el ENCARGADO
  lo gestiona manualmente cambiando el tipo a NORMAL y anotando
  el permiso en el campo observaciones. No hay forma de registrar
  que una parte de la jornada fue de un tipo y otra parte de otro.
MEJORA PROPUESTA PARA V2.0:
  Modelar tramos de jornada dentro del mismo dia. Cada tramo
  tendria tipo, hora inicio y hora fin propios. El fichaje diario
  pasaria a ser un contenedor de tramos en lugar de un registro
  unico con un solo tipo.
  Requiere rediseno del modelo de datos (DDL v6) y actualizacion
  de los endpoints E22, E23 y de la logica de calculo de
  jornadaEfectivaMinutos.
BENEFICIO:
  Reflejo exacto de jornadas mixtas sin depender de observaciones
  de texto libre. Mejora la precision de los informes y los saldos.

----------------------------------------------------
MEJORA M-007
----------------------------------------------------
AREA: Modelo de datos -- Trazabilidad de entrada y salida en Fichaje
DETECTADA: 21/03/2026 (sesion 18)
QUE FALTA EN V1.0:
  En v1.0 el campo usuario en Fichaje es unico para todo el
  registro. Si el ADMIN corrige solo la hora de salida de un
  fichaje, el campo usuario apunta al creador original del
  registro. No es posible distinguir si la intervencion manual
  afecto a la entrada, a la salida o a ambas. En el informe HTML
  el asterisco aparece en las dos columnas aunque solo se haya
  tocado una.
MEJORA PROPUESTA PARA V2.0:
  Anadir campos usuarioEntrada y usuarioSalida en la entidad
  Fichaje. Cada campo apunta al usuario que registro o modifico
  esa hora de forma independiente. El informe reflejaria el
  asterisco unicamente en la columna afectada.
  Requiere DDL v6 y actualizacion de E22 y E23.
BENEFICIO:
  Trazabilidad granular de quien registro cada hora del fichaje.
  El informe refleja con precision donde se produjo la intervencion
  manual.

----------------------------------------------------
MEJORA M-008
----------------------------------------------------
AREA: Modelo de datos -- Trazabilidad de inicio y fin de pausa
DETECTADA: 21/03/2026 (sesion 18)
QUE FALTA EN V1.0:
  En v1.0 el campo usuario en Pausa es unico para todo el
  registro. No es posible distinguir si la intervencion manual
  afecto al inicio de la pausa, al fin o a ambos. En el informe
  HTML el asterisco aparece en las dos columnas aunque solo se
  haya tocado una.
MEJORA PROPUESTA PARA V2.0:
  Anadir campos usuarioInicio y usuarioFin en la entidad Pausa.
  Cada campo apunta al usuario que registro o modifico esa hora
  de forma independiente. El informe reflejaria el asterisco
  unicamente en la columna afectada.
  Requiere DDL v6 y actualizacion de E27 y E28.
BENEFICIO:
  Trazabilidad granular de quien registro cada extremo de la
  pausa. Complementa M-007 para una auditoria completa de la
  jornada laboral.

MEJORA M-009
----------------------------------------------------
AREA: Operativa diaria -- Notificaciones automaticas
DETECTADA: 21/03/2026 (sesion 18)
QUE FALTA EN V1.0:
  En v1.0 el ENCARGADO tiene que consultar manualmente el informe
  o los endpoints de presencia para detectar ausencias
  injustificadas o fichajes incompletos al final del dia.
  ProcesoCierreDiario actua a las 23:55 sin avisar a nadie.
MEJORA PROPUESTA PARA V2.0:
  Notificacion push o email al ENCARGADO cuando ProcesoCierreDiario
  genera una o mas AUSENCIA_INJUSTIFICADA. La notificacion
  incluiria la lista de empleados afectados para que el ENCARGADO
  pueda revisar y corregir si procede al dia siguiente.
  Requiere integracion con servicio de notificaciones (FCM para
  Android o SMTP para email) y campo email o token FCM en la
  entidad Usuario.
BENEFICIO:
  El ENCARGADO es informado proactivamente de incidencias sin
  necesidad de consultar el sistema activamente cada noche.

MEJORA M-010
----------------------------------------------------
AREA: Informes -- Exportacion PDF desde informe HTML
DETECTADA: 21/03/2026 (sesion 18)
QUE FALTA EN V1.0:
  En v1.0 el informe HTML (E42, E43) y los informes PDF (E45-E47)
  son endpoints separados con logica de generacion independiente.
  No hay forma de exportar a PDF directamente desde el informe
  HTML que se esta visualizando en pantalla.
MEJORA PROPUESTA PARA V2.0:
  Boton Exportar a PDF dentro del informe HTML que llame a un
  endpoint de generacion PDF con los mismos parametros (empleadoId,
  desde, hasta) usando la misma logica de iText 7 de E45-E47.
  Unificaria la logica de informe en un unico servicio evitando
  duplicidad de codigo.
BENEFICIO:
  El usuario puede pasar de visualizar el informe en pantalla a
  descargarlo en PDF en un solo paso sin salir de la aplicacion.

MEJORA M-011
----------------------------------------------------
AREA: Informes -- Informe de pausas independiente
DETECTADA: 21/03/2026 (sesion 18)
QUE FALTA EN V1.0:
  En v1.0 el detalle de pausas aparece integrado dentro del
  informe de jornada (E42, E43). No existe un endpoint especifico
  para consultar solo las pausas de un empleado en un periodo,
  sin el contexto de la jornada completa.
MEJORA PROPUESTA PARA V2.0:
  Nuevo endpoint GET /api/v1/informes/pausas/{empleadoId} con
  parametros ?desde=date ?hasta=date ?tipo=tipoPausa ?formato=json|html.
  Devolveria solo la tabla de pausas con inicio, fin, duracion
  y motivo, filtrable por tipo de pausa (COMIDA, DESCANSO,
  AUSENCIA_RETRIBUIDA, OTROS).
BENEFICIO:
  Permite analizar patrones de pausas de forma independiente.
  Util para detectar pausas excesivas o irregulares sin tener
  que revisar el informe de jornada completo.

MEJORA M-012
----------------------------------------------------
AREA: Modelo de datos -- Origen del fichaje (planificado vs manual)
DETECTADA: 21/03/2026 (sesion 19, verificacion E42)
QUE FALTA EN V1.0:
  En v1.0 la entidad Fichaje no tiene ningun campo que indique
  si el registro fue generado por ProcesoCierreDiario a partir
  de una PlanificacionAusencia o creado directamente por el
  encargado como correccion o intervencion puntual. Ambos casos
  tienen usuario_id del encargado. El informe HTML no puede
  distinguirlos y muestra festivos y vacaciones planificadas
  como intervenciones manuales en la seccion "Intervenciones
  manuales" (ver L-007).
  Workaround en v1.0: consulta adicional a planificacion_ausencias
  por empleadoId y fecha en InformeService (Opcion B, ver D-028).
MEJORA PROPUESTA PARA V2.0:
  Anadir campo planificacion_id (FK nullable) en la tabla fichajes
  que apunte al registro de planificacion_ausencias del que
  procede el fichaje. ProcesoCierreDiario lo rellena al crear
  fichajes desde planificaciones. Si planificacion_id != NULL el
  fichaje es de origen planificado y no cuenta como intervencion
  manual en el informe. Si planificacion_id IS NULL y el usuario
  no es EMPLEADO ni terminal_service, si es intervencion manual.
  Requiere DDL v6, nueva FK en fichajes, y actualizacion de
  ProcesoCierreDiario.java e InformeService.java.
BENEFICIO:
  El informe distingue con precision entre registros planificados
  e intervenciones manuales reales. Elimina los falsos positivos
  de la seccion "Intervenciones manuales" sin queries adicionales.

MEJORA M-013
----------------------------------------------------
AREA: Modelo de datos -- Auditoria de intervenciones manuales
DETECTADA: 21/03/2026 (sesion 20)
QUE FALTA EN V1.0:
  En v1.0 cuando el ADMIN o el ENCARGADO crean o modifican un
  fichaje, una pausa o una ausencia de forma manual, el registro
  original se sobreescribe sin dejar rastro del estado previo.
  El campo usuario_id en las entidades Fichaje y Pausa indica
  quien fue el ultimo en tocar el registro, pero no guarda el
  valor anterior ni el momento exacto del cambio. Si hay una
  discrepancia no hay forma de reconstruir el historial de
  modificaciones ni saber que habia antes de la intervencion.
MEJORA PROPUESTA PARA V2.0:
  Nueva tabla auditoria_intervenciones que registra cada INSERT
  o UPDATE manual sobre fichajes, pausas y planificacion_ausencias.
  Las eliminaciones no aplican porque el modelo de v1.0 no expone
  endpoints DELETE sobre estos registros.
  Campos propuestos:
    - id (PK autoincremental)
    - tabla_afectada (fichajes | pausas | planificacion_ausencias)
    - registro_id (id del registro modificado o creado)
    - accion (INSERT | UPDATE)
    - usuario_id (quien ejecuto la accion)
    - fecha_hora (timestamp de la accion)
    - estado_anterior (JSON snapshot del registro antes del cambio,
      NULL si la accion es INSERT)
    - estado_nuevo (JSON snapshot del registro tras el cambio)
    - observacion (texto libre opcional para justificar la accion)
  El registro se crea en el service correspondiente justo antes
  de ejecutar el save() o merge() del registro afectado.
  Requiere DDL v6, nueva entidad JPA AuditoriaIntervencion y
  actualizacion de FichajeService, PausaService y AusenciaService.
BENEFICIO:
  Trazabilidad completa de todas las intervenciones manuales.
  El ADMIN puede reconstruir el historial exacto de cualquier
  registro, saber que habia antes de cada cambio y quien lo hizo.
  Complementa M-005 (historial de saldos) y M-012 (origen del
  fichaje) para una auditoria integral del sistema.

====================================================
LIMITACIONES CONOCIDAS DE V1.0
====================================================

Las siguientes limitaciones son consecuencia directa de las
mejoras pendientes. Se documentan aqui para que puedan
mencionarse en la memoria final del TFG como trabajo futuro.

L-001: El ENCARGADO no puede adelantar el cierre del dia.
  En v1.0 el cierre lo ejecuta ProcesoCierreDiario a las 23:55.
  El ENCARGADO puede seguir modificando registros del dia actual
  hasta esa hora aunque considere el dia cerrado.
  Solucion en M-002.

L-002: No hay trazabilidad de los cambios en saldos anuales.
  Si un saldo esta mal no hay forma de saber cuando ni por que
  cambio. Solucion en M-005.

L-003: El JWT de 12h no se puede renovar de forma transparente.
  Si expira durante la jornada el usuario debe hacer login de nuevo.
  Solucion en M-001.

L-004: No hay detalle de horas de permiso retribuido dentro del dia.
  Si un empleado tiene permiso retribuido parcial, solo se refleja
  en observaciones de texto libre. No hay desglose de tramos.
  Solucion en M-006.

L-005: El asterisco de intervencion manual en Fichaje no distingue
  si afecto a la entrada, a la salida o a ambas.
  En v1.0 el campo usuario es unico para todo el registro.
  Solucion en M-007.

L-006: El asterisco de intervencion manual en Pausa no distingue
  si afecto al inicio, al fin o a ambos.
  En v1.0 el campo usuario es unico para todo el registro.
  Solucion en M-008.

L-007: La seccion "Intervenciones manuales" del informe HTML puede
  mostrar falsos positivos. Festivos, vacaciones y otros tipos de
  ausencia creados por el encargado de forma planificada aparecen
  como intervenciones manuales porque en v1.0 no hay forma de
  distinguirlos de correcciones reales solo con el campo usuario_id.
  Workaround en v1.0: consulta a planificacion_ausencias (D-028).
  Solucion definitiva en M-012.
