FASE 2 — BLOQUE 1 — ENTIDADES JPA
====================================================

CAUSA RAÍZ COMÚN: las 7 entidades fueron creadas en Fase 0
como placeholders antes de que el diagrama de clases UML y el
modelo de datos estuvieran cerrados. El diagrama de clases
definitivo (staffflow_class_diagram.puml) se diseñó en la
Fase 1 (Bloque 2) junto con el DDL v4. Al llegar a la
implementación (Fase 2, Bloque 1) se detectó que las entidades
placeholder de Fase 0 no coincidían con el diagrama de clases
ni con el DDL definitivo. Se corrigieron todas antes de
continuar con DTOs y Repositories.

ORDEN CORRECTO aplicado en este proyecto:
  Diagrama de clases (Fase 1, Bloque 2)
  → DDL MySQL (Fase 1, Bloque 2)
  → Entidades JPA corregidas (Fase 2, Bloque 1)
  → DTOs → Repositories → Services

LECCIÓN DOCUMENTADA: en un proyecto con metodología iterativa,
los artefactos de implementación (entidades JPA, DTOs) deben
crearse SIEMPRE después de que el diagrama de clases y el diseño
estén cerrados. Crear entidades antes del diagrama de clases
multiplica el coste de corrección al propagar errores a todas
las capas posteriores.

DESVIACIÓN D-003
----------------------------------------------------
FASE/BLOQUE: Fase 2, Bloque 1
FICHERO: Fichaje.java
QUÉ ESTABA MAL:
  - @UniqueConstraint incorrecta: incluía (empleado_id, fecha, tipo)
    en lugar de solo (empleado_id, fecha)
  - Campo hora (LocalTime) en lugar de horaEntrada + horaSalida
    (LocalDateTime, nullable)
  - FK registradoPor en lugar de usuario (nombre del DDL)
  - Campo manual (Boolean) que no existe en el DDL
  - Faltaban: totalPausasMinutos, jornadaEfectivaMinutos, fechaCreacion
  - tipo length=20 insuficiente (AUSENCIA_INJUSTIFICADA = 22 chars)
POR QUÉ OCURRIÓ:
  La UniqueConstraint con tipo fue un error conceptual en el
  placeholder de Fase 0, previo al diagrama de clases definitivo:
  se pensó que podría haber múltiples tipos en el mismo día, pero
  el diagrama de clases (Fase 1, Bloque 2) y el modelo correcto
  establecen un único fichaje por empleado por día (RNF-I02,
  RD-ley 8/2019). El campo hora singular era un placeholder antes
  de que el diagrama definiera que el fichaje recoge horaEntrada
  Y horaSalida como LocalDateTime.
CORRECCIÓN:
  UniqueConstraint corregida a (empleado_id, fecha).
  hora → horaEntrada y horaSalida (ambos LocalDateTime nullable).
  FK renombrada a usuario. Campo manual eliminado.
  Añadidos totalPausasMinutos, jornadaEfectivaMinutos,
  fechaCreacion con @PrePersist. tipo length=30.
QUÉ IMPLICA:
  La UniqueConstraint incorrecta habría permitido dos fichajes del
  mismo empleado el mismo día con tipos distintos, violando el
  RD-ley 8/2019 (RNF-I02). horaEntrada y horaSalida nullable
  permite registrar ausencias sin horas (VACACIONES, BAJA_MEDICA...)
  en la misma tabla central. fechaCreacion permite detectar registros
  retroactivos (trazabilidad RNF-L02). jornadaEfectivaMinutos se
  calcula con Math.ceil (redondeo al alza, beneficia al empleado).

FASE/BLOQUE: Fase 2, Bloque 1

DESVIACIÓN D-005
----------------------------------------------------
FASE/BLOQUE: Fase 2, Bloque 1
FICHERO: PlanificacionAusencia.java
QUÉ ESTABA MAL:
  - fechaInicio + fechaFin (rango) en lugar de fecha (un día)
  - FK creadoPor en lugar de usuario
  - Campo procesado ausente (crítico para el @Scheduled)
  - Faltaban @UniqueConstraint en (empleado_id, fecha)
  - Faltaba @Index en (fecha, procesado)
  - Faltaba fechaCreacion
  - tipoAusencia length=20 insuficiente (DIA_LIBRE_COMPENSATORIO = 24 chars)
POR QUÉ OCURRIÓ:
  El modelo de rango (fechaInicio/fechaFin) era intuitivo
  visualmente y se usó en el placeholder de Fase 0 antes de que
  el diagrama de clases definitivo (Fase 1, Bloque 2) estableciera
  el modelo un-registro-por-día. La decisión de usar fecha única
  se tomó al diseñar el diagrama de clases y el DDL, pero no se
  trasladó a la entidad placeholder de Fase 0.
CORRECCIÓN:
  fechaInicio/fechaFin → fecha (un registro por día).
  FK renombrada a usuario. procesado añadido (Boolean, default false).
  @UniqueConstraint en (empleado_id, fecha).
  @Index en (fecha, procesado). fechaCreacion con @PrePersist.
  tipoAusencia length=25.
QUÉ IMPLICA:
  El modelo un-registro-por-día simplifica el proceso @Scheduled
  (00:01): busca WHERE fecha = HOY AND procesado = false, sin
  necesidad de expandir rangos. Si un empleado tiene una semana
  de vacaciones se crean 5 registros (uno por día laboral), lo que
  también facilita cancelar días individuales. procesado es la pieza
  central del proceso nocturno: sin él no hay idempotencia.

DESVIACIÓN D-013
----------------------------------------------------
FASE/BLOQUE: Fase 2, Bloque 1, detectada 10/03/2026
FICHERO: staffflow_v4_ddl_mysql.sql → staffflow_v5_ddl_mysql.sql
QUÉ ESTABA MAL:
  Todas las columnas id y FK del DDL v4 estaban declaradas como INT:
    configuracion_empresa.id
    usuarios.id
    empleados.id, usuario_id
    planificacion_ausencias.id, empleado_id, usuario_id
    fichajes.id, empleado_id, usuario_id
    pausas.id, empleado_id, usuario_id
    saldos_anuales.id, empleado_id
POR QUÉ OCURRIÓ:
  Las entidades JPA declaran sus PKs como Long (Java, 64 bits),
  que Hibernate mapea a BIGINT en SQL. El DDL v4 usaba INT (32 bits)
  por convención habitual en MySQL para IDs de tablas pequeñas,
  sin tener en cuenta que con ddl-auto:validate Hibernate exige
  coincidencia exacta de tipos. Al activar el perfil mysql por
  primera vez el arranque falló con SchemaManagementException.
CORRECCIÓN:
  Todas las columnas afectadas cambiadas de INT a BIGINT en MySQL
  mediante ALTER TABLE con FOREIGN_KEY_CHECKS=0 (necesario porque
  las FK impedían el cambio directo). DDL actualizado a v5 con
  los tipos correctos.
QUÉ IMPLICA:
  BIGINT es la elección correcta para PKs con Long en JPA.
  INT (32 bits) habría sido suficiente para el volumen de datos
  de StaffFlow, pero la convención de Spring Boot / Hibernate es
  Long/BIGINT y mantenerla evita este tipo de conflictos.
  El DDL v5 es el script de referencia para recrear la BD desde cero.
DOCUMENTOS ACTUALIZADOS:
  staffflow_v5_ddl_mysql.sql (nuevo nombre de referencia)
  StaffFlow_Desviaciones.txt (esta entrada)
  StaffFlow_Fase2.txt (sesión 10/03/2026)

DESVIACIÓN D-014
----------------------------------------------------
FASE/BLOQUE: Fase 2, Bloque 1, detectada 10/03/2026
FICHERO: Empleado.java
QUÉ ESTABA MAL:
  El campo pinTerminal estaba anotado sin columnDefinition:
    @Column(name = "pin_terminal", nullable = false, unique = true, length = 4)
  Hibernate mapea String con length=4 a VARCHAR(4) por defecto.
  El DDL tiene pin_terminal CHAR(4). Son tipos distintos para
  Hibernate con ddl-auto:validate, que falló con:
    found [char (Types#CHAR)], but expecting [varchar(4) (Types#VARCHAR)]
POR QUÉ OCURRIÓ:
  Al corregir Empleado.java en la sesión 07/03/2026 (D-002) se
  alinearon los campos con el DDL pero no se tuvo en cuenta que
  Hibernate trata CHAR y VARCHAR como tipos distintos en la
  validación estricta del schema, aunque en MySQL son compatibles.
CORRECCIÓN:
  Añadido columnDefinition = "CHAR(4)" en la anotación @Column
  del campo pinTerminal. Así Hibernate sabe que debe esperar CHAR
  y la validación pasa correctamente.
  Código resultante:
    @Column(name = "pin_terminal", nullable = false, unique = true,
            length = 4, columnDefinition = "CHAR(4)")
POR QUÉ CHAR Y NO VARCHAR:
  CHAR(4) es semánticamente correcto para el PIN: siempre tiene
  exactamente 4 dígitos. CHAR reserva espacio fijo, lo que es
  más eficiente para valores de longitud constante. VARCHAR añade
  overhead de longitud variable sin beneficio cuando la longitud
  es siempre la misma.
QUÉ IMPLICA:
  Solo impacto en la anotación de la entidad. La BD y el DDL
  no cambian (CHAR(4) ya era correcto en el DDL v4/v5).
DOCUMENTOS ACTUALIZADOS:
  Empleado.java (campo pinTerminal)
  StaffFlow_Desviaciones.txt (esta entrada)

DESVIACIÓN D-015
----------------------------------------------------
FASE/BLOQUE: Fase 2, Bloque 1, detectada 10/03/2026
FICHERO: pom.xml
QUÉ ESTABA MAL:
  El pom.xml no incluía la dependencia del conector JDBC de MySQL.
  Al arrancar con el perfil mysql el arranque fallaba con:
    Cannot load driver class: com.mysql.cj.jdbc.Driver
POR QUÉ OCURRIÓ:
  Spring Boot incluye H2 automáticamente con spring-boot-starter-data-jpa
  para el perfil de desarrollo, pero el conector MySQL no es transitivo
  y debe añadirse explícitamente. Durante el diseño de la Fase 1 y el
  Bloque 1 de la Fase 2 el proyecto solo arrancaba con perfil dev (H2),
  por lo que la ausencia del conector MySQL no se detectó hasta intentar
  arrancar con el perfil mysql por primera vez.
CORRECCIÓN:
  Añadida la dependencia en pom.xml:
    <dependency>
        <groupId>com.mysql</groupId>
        <artifactId>mysql-connector-j</artifactId>
        <scope>runtime</scope>
    </dependency>
  Spring Boot gestiona la versión automáticamente (9.6.0 en este caso).
  scope runtime es correcto: el conector solo se necesita en ejecución,
  no en compilación.
QUÉ IMPLICA:
  Sin esta dependencia el perfil mysql no puede arrancar en ningún
  entorno. Es imprescindible para el despliegue en producción.
  El perfil dev (H2) sigue funcionando sin ella.
DOCUMENTOS ACTUALIZADOS:
  pom.xml
  StaffFlow_Desviaciones.txt (esta entrada)

AFECTA A: StaffFlow_Fase2_v2.docx, README v3/v4
QUÉ ESTABA DOCUMENTADO:
  JUnit + JaCoCo con cobertura >= 70% en capa service.
  Aparecía en StaffFlow_Fase2_v2.docx (línea 12) y

  el tribunal: se aporta la cifra real obtenida junto con
  la justificación por criticidad del área testeada.
ALTERNATIVA DESCARTADA:
  Mantener el umbral >= 70% — implicaría testear controllers,
  DTOs y configuración para llegar al número, desviando esfuerzo
  de los servicios que realmente importan.

DESVIACIÓN D-010
----------------------------------------------------
FASE/BLOQUE: Fase 2, decisión tomada 10/03/2026
AFECTA A: StaffFlow_Fase2_v2.docx, diseño inicial de procesos
QUÉ ESTABA DOCUMENTADO:
  Tres procesos automáticos con @Scheduled:
    - ProcesoDiario @Scheduled 00:01
    - CierreDiario @Scheduled (implícito, fin de jornada)
    - CierreAnual @Scheduled 31-dic
CAMBIO DECIDIDO:
  ProcesoDiario permanece @Scheduled 00:01.
  CierreDiario pasa a botón manual ENCARGADO/ADMIN (E40).
  CierreAnual pasa a botón manual exclusivo ADMIN (E41).
  CierreDiario.java y CierreAnual.java como @Scheduled: ELIMINADOS.
POR QUÉ SE CAMBIA:
  CierreDiario: el encargado debe verificar que no hay fichajes
  incompletos antes de cerrar. Un @Scheduled que cierra sin
  verificación previa genera errores silenciosos en saldos_anuales

FASE/BLOQUE: Fase 2, Bloque 4, detectada 13/03/2026 (sesión 9 — verificación)
FICHERO: ConflictException.java (nuevo) · GlobalExceptionHandler.java
         UsuarioService.java · EmpleadoService.java
QUÉ ESTABA DOCUMENTADO:
  Los services usaban IllegalStateException para dos propósitos
  semánticamente distintos:
    (a) Recurso no encontrado en BD → correspondía HTTP 404
    (b) Dato ya registrado en otro recurso → correspondía HTTP 409
  GlobalExceptionHandler mapeaba IllegalStateException siempre a HTTP 404
  sin distinción de caso.
  El Javadoc de UsuarioService mencionaba explícitamente "HTTP 409
  preventivo" pero la implementación real producía 404 en ambos casos.
QUÉ SE ENCONTRÓ AL VERIFICAR:
  Durante la verificación completa de E08-E21 (sesión 9) se detectó:
    - E11 PATCH /usuarios/{id} con email duplicado → devolvía 404, no 409.
    - E18 PATCH /empleados/{id}/reactivar con empleado ya activo → devolvía 404, no 409.
  La herramienta de verificación fue Swagger UI y PowerShell (Invoke-WebRequest).
CORRECCIÓN APLICADA:
  1. ConflictException.java — nueva excepción custom en paquete exception.
     Extiende RuntimeException. Sin campos adicionales. Mensaje en constructor.
  2. GlobalExceptionHandler.java — nuevo handler añadido:
       @ExceptionHandler(ConflictException.class) → HTTP 409 CONFLICT
     Posición en el fichero: entre el handler 403 y el handler 423.
  3. UsuarioService.java — throws cambiados a ConflictException:
       crear(): username duplicado
       crear() y actualizar(): email duplicado
  4. EmpleadoService.java — throws cambiados a ConflictException:
       crear(): DNI duplicado
       crear(): NSS duplicado
       crear(): PIN duplicado
       crear(): NFC duplicado (solo si codigoNfc no es null)
       reactivar(): empleado ya activo
SEPARACIÓN DEFINITIVA DE EXCEPCIONES EN EL SISTEMA:
  IllegalStateException    → HTTP 404 Not Found
    Semántica: el recurso solicitado no existe en BD.
    Ejemplo: findById(id) no encuentra el registro.
  ConflictException        → HTTP 409 Conflict
    Semántica: el dato enviado viola una restricción de unicidad.
    Ejemplo: username ya registrado en otro usuario.
  IllegalArgumentException → HTTP 400 Bad Request
    Semántica: dato de entrada inválido según lógica de negocio.
    Ejemplo: password actual incorrecta en cambiar contraseña.
  PinBloqueadoException    → HTTP 423 Locked
    Semántica: terminal bloqueado por exceso de intentos PIN.
POR QUÉ ES CORRECTO:
  HTTP 404 y HTTP 409 son semánticas distintas que el cliente Android
  debe tratar de forma diferente:
    404 → el recurso referenciado no existe (el cliente debe revisar el id).
    409 → el dato enviado viola unicidad (el cliente debe cambiar el valor).
  Mezclar ambos en la misma excepción obliga al cliente a analizar el
  texto del mensaje para distinguir los casos, lo cual es frágil y
  viola el contrato del protocolo HTTP.
QUÉ IMPLICA:
  ConflictException es reutilizable en todos los services del Bloque 5
  en adelante para cualquier conflicto de unicidad:
    - FichajeService: UNIQUE(empleado_id, fecha) — fichaje duplicado
    - PausaService: pausa activa ya existente (WHERE hora_fin IS NULL)
