      empleadoRepository.findAllByActivoTrue() (verificar nombre exacto).

      Verificar con fichajeRepository.findByEmpleadoIdAndFecha().
      Si no existe: crear Fichaje con:
        tipo = TipoFichaje.AUSENCIA_INJUSTIFICADA

        horaEntrada = null
        horaSalida = null

        usuario = terminal_service (usuarioId=4, cargar desde repo)
      Guardar con fichajeRepository.save().

  TAREA B -- Procesado planificaciones pendientes:
    Buscar con ausenciaRepository.findPendientesByFechaLessThanEqual(
      LocalDate.now().plusDays(1)).
    Para cada PlanificacionAusencia:
      Si empleadoId == null (festivo global RF-26):
        Para cada empleado activo: crear fichaje si no existe ya.
      Si empleadoId != null (ausencia individual):
        Verificar que no existe ya fichaje para ese empleado/fecha.
        Si no existe: crear fichaje con TipoFichaje correspondiente.
    Mapeo TipoAusencia -> TipoFichaje (todos directos por nombre):
      FESTIVO_NACIONAL     -> TipoFichaje.FESTIVO_NACIONAL
      FESTIVO_LOCAL        -> TipoFichaje.FESTIVO_LOCAL
      VACACIONES           -> TipoFichaje.VACACIONES
      ASUNTO_PROPIO        -> TipoFichaje.ASUNTO_PROPIO
      PERMISO_RETRIBUIDO   -> TipoFichaje.PERMISO_RETRIBUIDO
      DIA_LIBRE_COMPENSATORIO -> TipoFichaje.DIA_LIBRE_COMPENSATORIO
    Marcar planificacion.setProcesado(true) y guardar.

  TAREA C -- Actualizacion saldos anuales:

    Para cada empleado activo: llamar a
      saldoService.recalcularParaProceso(empleado.getId(), anioActual).
    CRITICO: recalcular() en SaldoService es @Transactional y tiene
      logica correcta pero es llamado desde E40 (ADMIN). Para usarlo
      desde ProcesoCierreDiario hay dos opciones:

          metodo package-private o public en SaldoService llamado
          recalcularParaProceso(Long empleadoId, Integer anio) que
          no devuelve SaldoResponse (no hace falta el DTO) y que
          ProcesoCierreDiario puede llamar directamente.
        OPCION B: hacer public el metodo recalcular() existente y
          llamarlo desde ProcesoCierreDiario ignorando el SaldoResponse.
      Decidir con Santiago cual de las dos opciones aplicar.

      (proceso interno Spring, sin JWT ni rol).

PARTE 2 -- Restriccion de fecha para ENCARGADO (D-026)

  Services a modificar: FichajeService / PausaService / AusenciaService
  Endpoints afectados: E22 / E23 / E27 / E28 / E30 / E31
  E32 NO necesita restriccion (se protege solo con procesado=false).

  PATRON IDENTICO en los seis metodos:
    El controller extrae el rol del token JWT y lo pasa al service.
    En el service, al inicio de cada metodo afectado:

      if (esEncargado && !fecha.isEqual(LocalDate.now())) {
          throw new IllegalArgumentException(
              "El ENCARGADO solo puede gestionar registros del dia actual");
      }
      -> HTTP 400 via GlobalExceptionHandler

    Si rol == ADMIN: sin restriccion, pasa directo.

  COMO PASA EL ROL DEL CONTROLLER AL SERVICE:
    El controller recibe Authentication de Spring Security.
    Extrae el rol con:
      boolean esEncargado = authentication.getAuthorities().stream()
          .anyMatch(a -> a.getAuthority().equals("ROLE_ENCARGADO"));
    Lo pasa como parametro boolean al service.
    Alternativa: pasar el username y que el service resuelva el rol
    desde UsuarioRepository (mismo patron D-017 Opcion B).
    Decidir con Santiago cual de las dos opciones aplicar.
