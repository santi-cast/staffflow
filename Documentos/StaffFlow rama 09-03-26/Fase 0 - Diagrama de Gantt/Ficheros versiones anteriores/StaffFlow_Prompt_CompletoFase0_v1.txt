## BASE DE DATOS — 7 TABLAS

### Decisiones clave
- H2 embebido para entrega académica (sin instalación externa). MySQL 8.0 documentado como solución de producción futura. El script DDL MySQL completo va en un anexo de la memoria.
- Lógica de negocio 100% en servicios Spring Boot, nunca en BD (sin stored procedures ni triggers).
- ENUMs almacenados como VARCHAR en H2 mediante @Enumerated(EnumType.STRING). En producción MySQL se usarían tipos ENUM nativos.
- IDs independientes entre usuarios y empleados (no comparten secuencia AUTO_INCREMENT).

### configuracion_empresa — singleton (id=1 siempre)
Campos: id PK, nombre_empresa VARCHAR(100), cif VARCHAR(20) UNIQUE, direccion TEXT, email VARCHAR(100), telefono VARCHAR(20), logo_path VARCHAR(255)

### usuarios — autenticación y roles
Campos: id PK, username VARCHAR(50) UNIQUE, password_hash VARCHAR(255) BCrypt, email VARCHAR(100) UNIQUE
rol ENUM: ADMINISTRADOR | ENCARGADO | EMPLEADO
activo BOOLEAN DEFAULT TRUE, fecha_creacion TIMESTAMP
Índices: idx_username, idx_activo

### empleados — perfil laboral, relación 1:1 con usuarios
Campos: id PK (independiente de usuarios.id), usuario_id FK→usuarios UNIQUE
nombre VARCHAR(50), apellido1 VARCHAR(50), apellido2 VARCHAR(50)
dni VARCHAR(20) UNIQUE, nss VARCHAR(20) UNIQUE
fecha_alta DATE (para prorrateo vacaciones en alta a mitad de año)
categoria ENUM: ENCARGADO | REPONEDOR | CAJERO | LIMPIADOR | OTROS
jornada_semanal_horas INT DEFAULT 40
dias_vacaciones_anuales INT DEFAULT 22
dias_asuntos_propios_anuales INT DEFAULT 3
pin_terminal CHAR(4) UNIQUE — autenticación en terminal Android
codigo_nfc VARCHAR(50) UNIQUE NULL
activo BOOLEAN DEFAULT TRUE
Índices: idx_pin_terminal ⭐ CRÍTICO (búsqueda en cada fichaje), idx_dni, idx_activo

### planificacion_ausencias — ausencias futuras individuales y globales
Campos: id PK
empleado_id FK→empleados NULL PERMITIDO — NULL = festivo global (aplica a todos los empleados activos)
fecha DATE, tipo_ausencia ENUM, procesado BOOLEAN DEFAULT FALSE
usuario_id FK→usuarios (auditoría), observaciones TEXT, fecha_creacion TIMESTAMP
tipo_ausencia ENUM: FESTIVO_NACIONAL | FESTIVO_LOCAL | VACACIONES | ASUNTO_PROPIO | PERMISO_RETRIBUIDO | DIA_LIBRE_EXTRA
Índices: UNIQUE(empleado_id, fecha), idx(fecha, procesado) ⭐ CRÍTICO para proceso 00:01

### fichajes — registro diario consolidado, exactamente 1 por empleado por día
Campos: id PK, empleado_id FK→empleados, fecha DATE
tipo ENUM: NORMAL | FESTIVO_NACIONAL | FESTIVO_LOCAL | VACACIONES | ASUNTO_PROPIO | PERMISO_RETRIBUIDO | BAJA_MEDICA | DIA_LIBRE | DIA_LIBRE_COMPENSATORIO | AUSENCIA_INJUSTIFICADA
hora_entrada DATETIME NULL, hora_salida DATETIME NULL
total_pausas_minutos INT DEFAULT 0
jornada_efectiva_minutos INT DEFAULT 0 — calculado: Math.ceil(minutos_brutos - pausas_no_retribuidas)
usuario_id FK→usuarios (auditoría), observaciones TEXT, fecha_creacion TIMESTAMP
Índices: UNIQUE(empleado_id, fecha) ⭐, idx_empleado_id, idx_fecha, idx_tipo

### pausas — descansos dentro de la jornada, N pausas por día por empleado
Campos: id PK, empleado_id FK→empleados, fecha DATE
hora_inicio DATETIME, hora_fin DATETIME NULL — NULL significa pausa activa en este momento
duracion_minutos INT NULL — calculado con Math.floor al cerrar la pausa
tipo_pausa ENUM: COMIDA | DESCANSO | AUSENCIA_RETRIBUIDA | OTROS
usuario_id FK→usuarios, observaciones TEXT, fecha_creacion TIMESTAMP
Índices: idx(empleado_id, fecha) ⭐ join más frecuente
REGLA CLAVE: tipo AUSENCIA_RETRIBUIDA no descuenta de la jornada. Los demás tipos sí descuentan.

### saldos_anuales — saldo en tiempo real (año actual) e histórico (años cerrados)
Actualizado automáticamente cada noche a las 23:30 por CierreDiario.
Campos: id PK, empleado_id FK→empleados, anio INT
dias_trabajados INT DEFAULT 0
dias_baja_medica INT DEFAULT 0
dias_permiso_retribuido INT DEFAULT 0
dias_ausencia_injustificada INT DEFAULT 0
dias_vacaciones_derecho_anio INT
dias_vacaciones_pendientes_anio_anterior INT DEFAULT 0 — arrastre del año anterior
dias_vacaciones_consumidos INT DEFAULT 0
dias_vacaciones_disponibles INT — = derecho + pendientes - consumidos
dias_asuntos_propios_derecho_anio INT
dias_asuntos_propios_pendientes_anio_anterior INT DEFAULT 0
dias_asuntos_propios_consumidos INT DEFAULT 0
dias_asuntos_propios_disponibles INT
horas_ausencia_retribuida DECIMAL(10,2)
saldo_horas DECIMAL(10,2) — positivo = horas extra / negativo = déficit
calculado_hasta_fecha DATE NULL — GARANTÍA DE IDEMPOTENCIA del proceso automático
fecha_ultima_modificacion TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
Índices: UNIQUE(empleado_id, anio) ⭐, idx_anio

## SERVICIOS PRINCIPALES

### AuthService
Login con usuario+contraseña. Generación de JWT de acceso (1h) y refresh token (7 días). Validación y renovación de tokens.

### FichajeService
Lógica de fichaje de entrada:
  1. Busca empleado por PIN en idx_pin_terminal (<1ms)
  2. Verifica empleado activo
  3. Verifica que no existe fichaje para hoy (UNIQUE constraint)
  4. Crea registro con hora_entrada = NOW()

Lógica de fichaje de salida:
  1. Localiza fichaje del día del empleado
  2. Verifica hora_salida = NULL (no ha salido antes)
  3. Verifica que no hay pausa activa (hora_fin = NULL)
  4. Calcula: jornada_efectiva = Math.ceil(minutos_brutos - total_pausas_no_retribuidas)
  5. Actualiza fichaje con hora_salida y jornada calculada

Redondeo AL ALZA (Math.ceil) favorable al empleado — equivale a ~5h extra al año aprox.

### PausaService
Inicio de pausa: crea registro con hora_fin = NULL.
Fin de pausa: duracion = Math.floor(minutos) para no retribuidas, sin redondeo para AUSENCIA_RETRIBUIDA.
Math.floor en pausas + Math.ceil en jornada = redondeo híbrido siempre favorable al empleado.

### AusenciaService
Planifica ausencias individuales (empleado_id != NULL) y globales (empleado_id = NULL = festivo).
Cuando empleado_id es NULL, el servicio obtiene todos los empleados activos y crea un registro para cada uno al ser procesado.

### SaldoService — el más complejo del sistema
Actualiza saldos_anuales según tipo de fichaje del día.
Prorrateo vacaciones para alta a mitad de año: Math.ceil(22 × diasDesdeAlta / 365.0)
Endpoint de recálculo manual idempotente: POST /api/saldos/recalcular (solo ADMIN)
El recálculo lee todos los fichajes del año y reconstruye el saldo desde cero — útil si un proceso automático falló.

## PROCESOS AUTOMÁTICOS (@Scheduled)

### ProcesoDiario — @Scheduled(cron = "0 1 0 * * *") — 00:01 cada día
```
Consulta: SELECT * FROM planificacion_ausencias WHERE fecha = HOY AND procesado = FALSE
Para cada registro encontrado:
  Si empleado_id != NULL:
    Crea fichaje tipo ausencia para ese empleado (ignora si ya existe)
  Si empleado_id = NULL (festivo global):
    Obtiene todos los empleados activos
    Crea fichaje FESTIVO para cada uno (INSERT IGNORE si ya existe)
Marca todos los registros como procesado = TRUE
Log: "[ProcesoDiario] N fichajes creados para FECHA"
```

### CierreDiario — @Scheduled(cron = "0 30 23 * * *") — 23:30 cada día
```
Obtiene todos los fichajes del día
Para cada fichaje:
  Obtiene o crea registro saldos_anuales del empleado para el año actual
  Según tipo de fichaje actualiza contadores:
    NORMAL              → dias_trabajados++, recalcula δ saldo_horas
    VACACIONES          → dias_vacaciones_consumidos++, dias_trabajados++
    ASUNTO_PROPIO       → dias_asuntos_propios_consumidos++, dias_trabajados++
    PERMISO_RETRIBUIDO  → dias_permiso_retribuido++, dias_trabajados++
    BAJA_MEDICA         → dias_baja_medica++ (no penaliza saldo_horas)
    FESTIVO_*           → sin cambios en contadores
    AUSENCIA_INJUST.    → dias_ausencia_injustificada++, saldo_horas -= jornada_esperada
  Suma pausas AUSENCIA_RETRIBUIDA → actualiza horas_ausencia_retribuida
  Recalcula *_disponibles = derecho + pendientes - consumidos
  Actualiza calculado_hasta_fecha = hoy → guarda saldo
Log: "[CierreDiario] N empleados procesados — FECHA"
```

### CierreAnual — @Scheduled(cron = "0 59 23 31 12 *") — 31 dic 23:59
```
Obtiene todos los empleados activos
Para cada empleado:
  Lee saldo del año actual
  Calcula pendientes a arrastrar:
    vacaciones_pendientes = dias_vacaciones_disponibles del año que cierra
    asuntos_pendientes = dias_asuntos_propios_disponibles del año que cierra
  Crea registro saldo año nuevo:
    dias_* = 0 (reset)
    vacaciones_derecho = empleado.dias_vacaciones_anuales (con prorrateo si alta reciente)
    vacaciones_pendientes_anio_anterior = vacaciones_pendientes (arrastre)
    vacaciones_disponibles = derecho + arrastre
    saldo_horas = 0 (reset)
Log: "[CierreAnual YYYY] N saldos creados para YYYY+1"
```

## DECISIONES ARQUITECTÓNICAS DOCUMENTADAS

### Decisión 1: IDs independientes entre usuarios y empleados
Las tablas usuarios y empleados tienen secuencias AUTO_INCREMENT independientes y se relacionan por FK. Permite que un ADMINISTRADOR exista sin tener perfil de empleado. Estándar de la industria: las relaciones se expresan con FK, no con coincidencia de IDs.

### Decisión 2: Lógica de negocio exclusivamente en Spring Boot
Sin stored procedures, triggers ni funciones almacenadas en BD. Toda la lógica reside en los servicios. Beneficios: 100% testeable con JUnit, portable entre motores de BD, versionado en Git junto con el código.

### Decisión 3: JWT stateless sin sesiones en servidor
Token JWT firmado con clave secreta, validado en cada petición por JwtAuthFilter. Sin estado en servidor. Si en el futuro se escala horizontalmente, no hay sesiones que sincronizar. Contexto: para una PYME con 50 empleados, la gestión de sesiones sería over-engineering.

### Decisión 4: empleado_id = NULL para festivos globales
Un registro en planificacion_ausencias con empleado_id = NULL representa un festivo que aplica a todos. Unifica planificación individual y global en un único endpoint, servicio y proceso. La restricción UNIQUE(empleado_id, fecha) permite múltiples NULL en la misma fecha (comportamiento estándar de índices únicos con NULLs en MySQL).

### Decisión 5: @Scheduled embebido en Spring Boot
Sin cron del SO ni sistemas de colas (RabbitMQ, Kafka). Los procesos automáticos reutilizan los mismos servicios que usa la API. Limitación reconocida: ejecución duplicada en escalado horizontal — mitigación futura con ShedLock (2 líneas de cambio).

## RIESGOS IDENTIFICADOS Y MITIGACIÓN

### Riesgo 1 — Inconsistencia en saldos por proceso automático fallido (Impacto: ALTO)
Mitigación:
- Campo calculado_hasta_fecha garantiza idempotencia: el proceso salta empleados ya procesados ese día
- Transacciones @Transactional independientes por empleado dentro del loop
- Endpoint POST /api/saldos/recalcular (solo ADMIN) permite reconstrucción completa desde cero
- Logging estructurado: inicio + fin + N empleados procesados en cada ejecución

### Riesgo 2 — Bug silencioso en cálculos de jornada acumulando error pequeño diariamente (Impacto: ALTO)
Mitigación:
- @ParameterizedTest con tabla de entrada/salida para todos los tipos de fichaje y casos límite
- Casos cubiertos: redondeo en 0.1 minutos, pausa retribuida no descontada, festivo en semana, alta a mitad de año, año bisiesto
- Recálculo idempotente como validación cruzada: proceso automático vs recálculo manual deben ser idénticos

### Riesgo 3 — Fuerza bruta sobre PINs (solo 10.000 combinaciones posibles) (Impacto: MEDIO)
Mitigación:
- Bloqueo tras 5 intentos fallidos: HTTP 429 durante 60 segundos
- El PIN solo permite generar fichajes, no acceder a funcionalidades de gestión ni datos de otros empleados

### Riesgo 4 — Token JWT del terminal comprometido (Impacto: MEDIO)
Mitigación:
- Token de acceso con expiración 1 hora
- HTTPS obligatorio en producción (Nginx + Let's Encrypt)
- El token pertenece al encargado, no a un empleado individual

### Riesgo 5 — Desajuste entre H2 (desarrollo) y MySQL (producción) (Impacto: BAJO)
Mitigación:
- H2 configurado con MODE=MySQL en la cadena de conexión
- ddl-auto: validate en producción (Hibernate valida el esquema en cada arranque)
- Script DDL MySQL probado independientemente en MySQL Workbench
