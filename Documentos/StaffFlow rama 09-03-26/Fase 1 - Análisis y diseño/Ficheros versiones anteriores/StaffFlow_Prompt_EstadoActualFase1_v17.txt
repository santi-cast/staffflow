  contexto de este TFG. Al expirar el token el usuario realiza

====================================================
ACTORES DEL SISTEMA
====================================================

ADMIN       Acceso total. Gestiona empresa, usuarios, empleados y operativa.
            NO tiene perfil de empleado — nunca ficha desde el terminal.

ENCARGADO   Gestiona la operativa diaria. SÍ tiene perfil de empleado.
            Ficha con PIN. No gestiona configuración ni usuarios.

EMPLEADO    Solo accede a sus propios datos. SÍ tiene perfil de empleado.
            Ficha con PIN. Spring Security bloquea acceso a datos ajenos (HTTP 403).

Separación usuarios/empleados:
  usuarios  → autenticación y rol (quién entra y con qué permisos)
  empleados → perfil laboral (datos como trabajador)
  ADMIN     → solo en usuarios (sin perfil de empleado)
  ENCARGADO / EMPLEADO → en ambas tablas

====================================================
DECISIONES DE DISEÑO CERRADAS
====================================================

1.  Registros inmutables
    Fichajes y pausas nunca se borran. Los errores se corrigen
    mediante modificación con observaciones obligatorias.
    Cumple RD-ley 8/2019 (RNF-L01, RNF-L02).

2.  Eliminación de ausencias solo si no procesadas
    Solo se puede eliminar una ausencia con procesado = false.
    Si ya está procesada, el fichaje existe y se modifica.

3.  Filtros GET opcionales y combinables
    Todos los endpoints GET aceptan parámetros opcionales
    que se pueden combinar libremente.

4.  Bajas lógicas (nunca DELETE)
    Empleados y usuarios se desactivan (activo = false).
    El historial queda intacto e integridad referencial preservada.

5.  Informes imprimibles vía HTML
    GET de informes acepta ?formato=html.
    Backend devuelve HTML limpio. Android imprime con
    PrintManager + WebView (sin dependencias externas de PDF).

6.  Informes firmables en PDF
    RF-38, RF-39, RF-40 generan PDF con iText 7.
    Son los únicos que requieren firma física del empleado.

7.  Ausencias: planificadas vs justificadas imprevistas
    Planificada → planificacion_ausencias (conocida con antelación)
    Justificada imprevista → fichaje manual con tipo correcto
    (BAJA_MEDICA, PERMISO_RETRIBUIDO...). NO va a planificacion.

8.  Terminal PIN vs fichaje manual
    Terminal → solo el propio empleado (RF-46..RF-49)
    Manual   → ADMIN o ENCARGADO ficha a cualquier empleado (RF-17)
    ADMIN nunca ficha desde el terminal (no tiene perfil empleado)

9.  Recuperación de contraseña
    email → token opaco 30 min → enlace reset → token invalidado
    Campos reset_token y reset_token_expiry en tabla usuarios.

10. Acceso a datos por rol
    ADMIN / ENCARGADO → datos de cualquier empleado
    EMPLEADO → Spring Security inyecta su ID. HTTP 403 si no coincide.

11. Pantalla standby del terminal
    Muestra hora en tiempo real + campo PIN siempre visible.
    Retorno automático a standby tras fichaje. (RNF-U01..U04)

12. Saldo anual — desglose completo siempre
    Vacaciones:     pendientes_año_ant | derecho_año | consumidos | disponibles
    Asuntos propios: ídem
    Horas:          trabajadas | esperadas | diferencia
    Aplica a RF-35, RF-36, RF-53 y PDF de RF-39, RF-40.

13. DIA_LIBRE_COMPENSATORIO — valor unificado
    Cubre dos situaciones con idéntico tratamiento contable:
    a) Día libre para compensar saldo_horas positivo acumulado
    b) Día libre por haber trabajado un festivo
    Aparece en TipoFichaje y TipoAusencia.

14. Visibilidad datos por rol
    ADMIN / ENCARGADO → todos los empleados
    EMPLEADO → solo los suyos (Spring Security, HTTP 403 si otro ID)

15. Umbrales de rendimiento en P95
    P95 = el 95% de las peticiones cumplen el umbral.
    Estándar de industria, defendible ante el tribunal.

16. Bloqueo PIN por dispositivo (no por empleado)
    Evita que el atacante eluda el límite rotando entre empleados.
    El empleado legítimo raramente falla 5 veces seguidas.

17. Observabilidad en dos capas
    BD: campo usuario_id en tablas operativas (quién hizo qué)
    Servidor: logs SLF4J/Logback por nivel (INFO, WARN, ERROR)
    Juntos forman un sistema de trazabilidad completo.

18. Sin refresh token en v1.0 — decisión deliberada y documentada
    JWT 8h cubre una jornada laboral completa sin interrupciones.
    Se evaluó el modelo access token (1h) + refresh token (7d) y se descartó:
    - Motivo técnico: el refresh token resuelve mantener sesión activa cuando
      el access token expira durante el uso. En StaffFlow ese problema no existe:
      el encargado hace login al inicio del turno y 8h cubren toda la jornada.
    - Motivo de alcance: requiere endpoint adicional, tabla en BD con índices
      y proceso de limpieza, lógica de rotación (cada uso emite token nuevo e
      invalida el anterior) y lógica de revocación (logout debe invalidar en BD,
      eliminando el beneficio stateless del JWT). Complejidad real sin beneficio
      práctico en un sistema de gestión laboral de uso diario.
    Si el tribunal pregunta: "Se valoró la alternativa y se descartó
    conscientemente por añadir complejidad sin beneficio en el contexto de uso
    del sistema. La decisión está documentada en el diseño." El hecho de haberla
    documentado la convierte en decisión de arquitectura, no en una omisión.

19. CategoriaEmpleado es informativo
    No afecta permisos (que dependen del rol en usuarios).
    Útil para filtros y exportaciones. Extensible en v2.0.

20. Swagger UI como documentación oficial de la API
    SpringDoc OpenAPI genera Swagger UI automáticamente desde
    los @RestController. Las capturas son la documentación de la
    API en la memoria del TFG (http://localhost:8080/swagger-ui.html).

21. Separación autenticación PIN / JWT — decisión de seguridad explícita
    El PIN de 4 dígitos es identificación rápida para fichaje en dispositivo
    compartido, NO autenticación completa. Dos capas claramente separadas:
    - PIN sin JWT: solo los 4 endpoints de /api/v1/terminal/ (entrada, salida,
      pausa/iniciar, pausa/finalizar). Operaciones de fichaje únicamente.
    - JWT (usuario+contraseña): todos los endpoints /me y el resto de la API.
      Acceso a saldos, historial, perfil, ausencias planificadas.
    Garantía: un PIN conocido por un compañero no permite acceder a datos
    personales ni información sensible.
    Extensibilidad: campo codigo_nfc ya previsto en BD para sustituir PIN por
    NFC en el futuro sin cambiar la separación de responsabilidades.
    Si el tribunal pregunta: "PIN resuelve velocidad de fichaje en terminal
    compartido. JWT resuelve acceso seguro a datos personales desde dispositivo
    propio. Son problemas distintos que requieren soluciones distintas." 

21. Jornada: dos campos complementarios en empleados
    jornada_semanal_horas (40): dato contractual introducido por ADMIN
    jornada_diaria_minutos (480): usado internamente en cálculos de
    saldo de horas. No son redundantes: expresan conceptos distintos.
