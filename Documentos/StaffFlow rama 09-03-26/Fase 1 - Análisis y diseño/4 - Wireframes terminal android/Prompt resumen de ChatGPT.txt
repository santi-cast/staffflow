====================================================
ARQUITECTURA ANDROID
====================================================

Arquitectura elegida:

Single Activity + Navigation Component + Fragments

Estructura:

MainActivity
  └ NavHostFragment
       ├ TerminalFragment (P01)
       ├ LoginFragment (P02)
       ├ RecoveryFragment (P03)
       ├ ResetPasswordFragment (P05)
       ├ HomeEmpleadoFragment (P09)
       ├ ParteDiarioFragment (P17)
       ├ EmpleadosFragment (P13)
       └ resto de fragments

Decisiones clave:

• El terminal (P01) es la pantalla raíz pública de la app.
• El Navigation Drawer solo aparece en la zona autenticada.
• El menú del Drawer se carga dinámicamente según rol.
• El destino inicial tras login depende del rol:
    ADMIN → P13 (Empleados)
    ENCARGADO → P17 (Parte diario)
    EMPLEADO → P09 (Mi saldo)

Persistencia de sesión:

JWT guardado en DataStore (no SharedPreferences).

Si cualquier endpoint devuelve HTTP 401:

DataStore.clear()
→ ocultar Drawer
→ navegar a P02 (Login)
→ mostrar snackbar "La sesión ha caducado"

Deep links:

El enlace de recuperación de contraseña abre directamente
ResetPasswordFragment mediante configuración en nav_graph.xml.

====================================================
DECISIONES DE DISEÑO CLAVE
====================================================

Terminal como raíz pública

El dispositivo donde se ejecuta la app puede estar colocado
como terminal físico de fichaje. Por ello la pantalla inicial
es siempre el terminal PIN.

Desde esa pantalla existen dos modos de uso:

1. Fichaje rápido mediante PIN
2. Acceso completo a la aplicación mediante login

Esto evita exponer menús administrativos en dispositivos
compartidos.

Flujo de pausa simplificado

Se eliminó el concepto de "modo pausa".
P01 muestra tres acciones explícitas:

• Fichar
• Iniciar pausa
• Finalizar pausa

Registro manual unificado

La pantalla P20 es reutilizable:

FormFichajeFragment

Dos variantes:
• fichaje manual
• pausa manual

El argumento de navegación determina la variante.

PUT vs PATCH

PUT → edición completa desde formulario
PATCH → cambios parciales de estado

Ejemplo:
PATCH /empleados/{id}/baja
PATCH /empleados/{id}/reactivar

====================================================
ESTADOS UX DOCUMENTADOS
====================================================

Estados de carga

CircularProgressIndicator en carga inicial.

Listas usan skeleton loading.

Estados de error

Mensaje descriptivo + botón "Reintentar".

Estados vacíos

Icono + mensaje + acción "Limpiar filtros"
si hay filtros activos.

Feedback terminal

Pantalla P06 muestra confirmación durante 3 segundos
y vuelve automáticamente a standby.

OBJETIVO DEL BLOQUE

El objetivo de este bloque es que:
