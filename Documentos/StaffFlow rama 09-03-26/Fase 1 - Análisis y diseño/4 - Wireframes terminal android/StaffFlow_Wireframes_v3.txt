====================================================
1. INTRODUCCIÓN Y CAMBIOS
====================================================

Este documento especifica las 24 pantallas de la aplicación Android de
StaffFlow. Cada pantalla incluye wireframe ASCII, endpoints que consume,
comportamiento, estados de carga/error/vacío y decisiones de diseño.

--- Cambios v2.0 ---

1. Métodos HTTP corregidos
   Todos los endpoints de edición usan PUT en wireframes, consistente con
   la especificación de endpoints. PUT envía el recurso completo.

2. Comportamiento global: sesión expirada
   Si cualquier endpoint autenticado devuelve HTTP 401 por token expirado
   o inválido, la app limpia DataStore, oculta el Drawer y redirige a P02
   (Login) mostrando el snackbar "La sesión ha caducado".

3. Drawer: sección Ajustes
   Cambiar contraseña y Cerrar sesión se agrupan bajo la sección
   "Ajustes" en el Drawer, en lugar de aparecer como entradas sueltas.

4. Estados vacíos
   Todas las pantallas con listas documentan su estado vacío:
   icono + mensaje descriptivo + acción secundaria "Limpiar filtros"
   cuando hay filtros activos.

5. Estados de carga y error
   Pantallas con llamadas a la API documentan: CircularProgressIndicator
   (carga inicial) y mensaje + botón Reintentar (error de red).
   Las listas usan skeleton en lugar de spinner.

6. Flujo pausa sin "modo"
   P01 expone tres botones explícitos: Fichar, Iniciar pausa, Finalizar
   pausa. Elimina el concepto abstracto de "modo pausa activo".

7. Tabla de permisos por rol
   Sección 5 añadida al final con matriz completa de visibilidad por rol
   (EMPLEADO / ENCARGADO / ADMIN).

--- Cambios v3.0 ---

A. PATCH para baja/reactivar
   PUT era incorrecto semánticamente para /baja y /reactivar. Son cambios
   parciales de estado (un campo: activo), no sustitución del recurso
   completo. Regla cerrada: PUT para formularios completos de edición ·
   PATCH para cambios de estado. Corregido en P14.
   Consistente con StaffFlow_Endpoints_v3.docx que ya tenía PATCH.

B. P20 formulación explícita
   La pantalla P20 se describe ahora explícitamente como "pantalla
   reutilizable de registro manual con dos variantes: fichaje manual y
   pausa manual". El argumento de navegación determina la variante
   activa al abrir el Fragment.

C. Ejemplo P13 corregido
   La fila de ejemplo "María López · Admin · Activo" se ha cambiado a
   "Administrativo · Activo". En el modelo de StaffFlow, ADMIN es un
   rol de usuario, no una categoría de empleado. Las categorías de
   empleado son: Operario, Administrativo, Técnico, Encargado, Otro.

  Endpoints:    POST /api/v1/terminal/entrada
                POST /api/v1/terminal/salida
                POST /api/v1/terminal/pausa/iniciar
                POST /api/v1/terminal/pausa/finalizar
  Autenticación: Sin JWT — identificación por PIN de 4 dígitos (decisión nº22)
  Comportamiento: Al 4º dígito → llamada automática. Botón activo determina
                  operación: Fichar (entrada/salida automática), Iniciar pausa
                  → P07 + PIN, Finalizar pausa → PIN directo.
                  OK → P06 (3s) → retorno automático. Error → vibración +
                  mensaje 2s + limpia dots.
  Errores:       PIN no reconocido | Ya has fichado hoy | Cierra la pausa
                  activa antes | HTTP 423 → dispositivo bloqueado
  Decisión nº23: Terminal como raíz pública sin Drawer. Tres acciones
                  explícitas evitan el concepto abstracto de "modo pausa".
  Estado vacío / carga: No aplica — pantalla siempre activa en standby

====================================================
4. DECISIONES TÉCNICAS DE NAVEGACIÓN ANDROID
====================================================

Decisiones tomadas en Fase 1 que condicionan la implementación de Fase 2.

nº24-A — Single Activity + Navigation Component
  MainActivity aloja un NavHostFragment. Cada pantalla es un Fragment.
  La navegación se gestiona mediante Navigation Component y nav_graph.xml.
  Reduce la complejidad del back stack y centraliza la navegación.
  Estándar recomendado por Google desde 2018.

nº24-B — Drawer en MainActivity, menú dinámico por rol
  El Drawer no vive en cada Fragment sino en MainActivity.
  Los Fragments públicos (TerminalFragment, LoginFragment, RecoveryFragment,
  ResetPasswordFragment) están marcados como destinos de nivel superior sin Drawer.
  Los autenticados lo activan automáticamente.
  El menú se infla según el rol del JWT.

nº24-C — JWT en DataStore + deep links para P05
  El token JWT se persiste en Preferences DataStore (no SharedPreferences, deprecated).
  Al arrancar se lee el token: si existe y no ha expirado (8h, decisión nº18)
  se navega directamente al destino por rol.
  El deep link de restablecimiento de contraseña se configura en nav_graph.xml
  apuntando a ResetPasswordFragment sin Activity adicional.

5. PERMISOS POR ROL
====================================================

Matriz de visibilidad y permisos. Cada pantalla y decisión de este
documento es consistente con estas reglas.

  Funcionalidad                                EMPLEADO  ENCARGADO  ADMIN
  ─────────────────────────────────────────    ────────  ─────────  ──────
  Mis datos propios (perfil, saldo,            ✓ lect.   ✓ lect.    ✓ lect.
    fichajes, ausencias)
  Parte diario (todos los empleados)           ✗         ✓          ✓
  Gestión empleados (crear, editar, baja)      ✗         ✓          ✓
  Fichajes manuales (crear, editar)            ✗         ✓          ✓
  Pausas manuales (crear, editar)              ✗         ✓          ✓
  Ausencias planificadas (crear, editar, borr) ✗         ✓          ✓
  Saldos globales (todos los empleados)        ✗         ✓ lect.    ✓
  Informes (horas, vacaciones)                 ✗         ✓          ✓
  Recalcular saldos                            ✗         ✗          ✓
  Ver PIN de empleado                          ✗         ✗          ✓
  Gestión de usuarios                          ✗         ✗          ✓
  Configuración empresa                        ✗         ✗          ✓

Regla general:
  EMPLEADO    → solo accede a sus propios datos en modo lectura.
  ENCARGADO   → gestiona la operativa diaria. No puede crear usuarios
                ni configurar la empresa.
  ADMIN       → acceso total.
  El backend valida el rol en cada endpoint mediante @PreAuthorize.
  La app solo controla visibilidad de UI — el backend es la autoridad.
